Option 1 (best): Move the SEO interceptor to run inside the SPA fallback only

If setupVite() installs (a) Vite middleware and (b) the SPA catch-all route, you can still get correct behavior by ensuring the SEO interceptor does not globally wrap responses. Instead:

Do not register SEO interceptor as app.use(seoInterceptor)

Instead, run SEO logic only when serving HTML inside the SPA fallback handler.

Why this works:

Module requests (/src/*, /@vite/*) are handled by vite.middlewares and never reach the SPA fallback.

Only “real” page navigations reach the fallback handler, where it’s safe to modify HTML.

Net effect: you don’t need to insert middleware between Vite and fallback at all.

What to tell your AI to implement

Remove global res.write/res.end wrapping middleware.

Implement SEO HTML rewriting where index.html is being served in the fallback route.

This is the cleanest fix and avoids touching server/vite.ts.

Option 2: Register API routes before calling setupVite()

If your immediate need is to get API routes in the right place, you can:

app.use("/api", apiRouter);
setupVite(app);

This places API routes before Vite middleware and before fallback, which is usually fine.

But it does not solve “SEO interceptor between Vite and fallback” unless you implement SEO rewriting as in Option 1.

Option 3 (most invasive): Bypass setupVite() and create Vite server in server/index.ts

This gives full ordering control, but it’s more work and easier to break.

Only do this if:

setupVite() is fundamentally incompatible with your needs

or you need additional Vite customization and can’t modify that file

What I recommend for you

Option 1. It directly matches the root cause: your current SEO interceptor wrapping res.write/res.end is interfering with streamed module responses. If you only rewrite HTML at the point you generate/send HTML, you eliminate the risk entirely.

Copy/paste prompt for your AI (non-invasive fix)

Use this to keep it focused and stop it from rewriting your server architecture:

Do NOT bypass setupVite and do NOT globally wrap res.write/res.end.

Because setupVite registers vite.middlewares and the SPA fallback together, implement SEO interception only inside the SPA fallback that serves index.html.
