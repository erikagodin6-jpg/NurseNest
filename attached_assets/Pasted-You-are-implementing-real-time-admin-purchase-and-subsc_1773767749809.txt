You are implementing real-time admin purchase and subscriber notifications for the NurseNest production codebase.

GOAL

Notify me immediately by email and optionally SMS whenever important commercial events happen on the site.

I want notifications for:
- new paid subscriber
- one-time purchase
- subscription renewal payment
- failed payment
- subscription cancellation
- optional: new free account signup

DELIVERY CHANNELS
- Email notifications to my configured admin email
- SMS notifications to my configured phone number
- Admin settings to enable/disable each notification type

PRIMARY REQUIREMENTS

Build this using a robust production-safe event architecture.

Use:
- Stripe webhooks for Stripe payment/subscription events
- app-level events for non-Stripe account signup if applicable

DO NOT:
- do not rely on frontend-only events
- do not send notifications before payment is confirmed
- do not create duplicate notifications for the same event
- do not hardcode secrets in frontend code
- do not make SMS mandatory if not configured
- do not spam multiple messages for one purchase

==================================================
PART 1 — EVENTS TO SUPPORT
==================================================

Implement notification handling for these events:

1. New paid subscriber
Trigger when a subscription is successfully created and payment is confirmed.

2. One-time purchase
Trigger when a one-time payment/check-out completes successfully.

3. Subscription renewal
Trigger when a recurring invoice is paid.

4. Failed payment
Trigger when a recurring payment fails.

5. Subscription cancelled
Trigger when a subscription is cancelled or set to cancel.

6. Optional free signup
Trigger when a user account is created without purchase, if the app already tracks this cleanly.

==================================================
PART 2 — WEBHOOK-BASED ARCHITECTURE
==================================================

Use Stripe webhooks as the source of truth for paid events.

Implement or audit the webhook endpoint so it correctly handles:
- checkout.session.completed
- customer.subscription.created
- customer.subscription.updated
- invoice.paid
- invoice.payment_failed

Use Stripe signature verification.
Use idempotency/event logging so the same Stripe event does not send duplicate notifications.

Create a durable event-processing flow:
- receive webhook
- verify signature
- identify event type
- map event to internal commercial event
- dedupe
- persist event log
- send notifications
- store send status/result

==================================================
PART 3 — EMAIL NOTIFICATIONS
==================================================

Implement email notifications first because they are essential.

Use the project’s existing email provider if one already exists.
If none exists, add a production-suitable provider such as Resend, Postmark, or SendGrid.

Create admin notification emails with clean formatting and useful details.

For each event email, include:
- event type
- customer email
- product name / plan name
- amount paid
- currency
- subscription interval if applicable
- date/time
- Stripe/customer/order reference if available
- direct admin link to view the user/order/subscription if such an admin page exists

Subject line examples:
- New Subscriber: NurseNest RN Monthly
- New Purchase: NCLEX Bundle
- Renewal Payment Received: NurseNest Subscription
- Payment Failed: Subscription Renewal
- Subscription Cancelled: NurseNest Subscription

==================================================
PART 4 — SMS NOTIFICATIONS
==================================================

Add optional SMS notifications using Twilio or the site’s existing SMS provider if already present.

SMS should be concise and reserved for high-signal events.

Default SMS events:
- new paid subscriber
- one-time purchase
- failed payment

Optional toggle for:
- renewal
- cancellation
- free signup

Example SMS:
- New subscriber: erika@example.com bought NurseNest RN Monthly for $19.99.
- New purchase: erika@example.com bought NCLEX Bundle for $29.99.
- Payment failed: renewal failed for erika@example.com.

If SMS credentials are not configured:
- do not break the app
- disable SMS gracefully
- show clear admin setting state

==================================================
PART 5 — ADMIN NOTIFICATION SETTINGS
==================================================

Build an admin settings area for commercial notifications.

Allow configuration of:
- notification email address
- notification phone number
- email enabled on/off
- SMS enabled on/off
- per-event toggles:
  - new subscriber
  - one-time purchase
  - renewal
  - payment failed
  - cancellation
  - free signup
- test email button
- test SMS button

Store settings securely in the backend/database or existing secure config system.

Do not expose secrets or phone/email settings insecurely.

==================================================
PART 6 — DEDUPLICATION / RELIABILITY
==================================================

Prevent duplicate sends.

Implement:
- processed webhook event table or equivalent
- unique event ID tracking
- notification delivery log
- retry-safe logic

If the same Stripe event is received more than once:
- do not send duplicate notifications

Log:
- event ID
- event type
- source
- notification channel
- send result
- timestamp
- any error message

==================================================
PART 7 — INTERNAL DATA MAPPING
==================================================

Map Stripe objects to useful internal business labels.

When possible, include:
- user name
- user email
- product or subscription plan name
- tier
- billing interval
- amount
- country if relevant
- source site/brand if multiple properties share infra

If metadata already exists in Stripe Checkout or subscriptions, use it.
If missing, improve metadata creation so future notifications are richer.

==================================================
PART 8 — SECURITY
==================================================

Requirements:
- verify Stripe webhook signatures correctly
- keep API keys/server secrets server-side only
- sanitize/log safely
- do not expose customer PII in frontend
- secure admin settings routes
- admin-only access for notification configuration

Implement this cleanly and production-safely.
