SECTION 1 — INVESTIGATION PHASE (MANDATORY)
==================================================

Audit the entire system and determine exactly where the failures originate.

Investigate all of the following:

A. BUILD / PUBLISH PIPELINE
- inspect the exact publish/build command
- identify every step executed during deploy
- identify anything non-essential happening during publish
- detect migrations, backfills, seed jobs, generators, sitemap jobs, translation jobs, image/PDF generation, content audits, cache rebuilds, or other heavy tasks running during deploy
- measure which step consumes the most time
- isolate dependency install time vs app build time vs startup time
- identify repeated work that should not happen every deploy
- identify whether generated assets/content are bloating the build context
- inspect whether too many routes, locale files, or content files are included in the build
- inspect whether source maps, debug artifacts, or heavy static assets are inflating build time
- inspect whether the app is doing full invalidation instead of incremental builds

B. RUNTIME MEMORY / SERVER STABILITY
- inspect memory use across the runtime
- identify high-memory code paths
- identify whether the server is restarting, being killed, hanging, or timing out
- inspect startup memory usage vs steady-state memory vs peak memory
- identify leaks, unbounded caches, large in-memory objects, huge serialized payloads, repeated content hydration, repeated translations loaded into memory, or oversized API responses
- identify whether heavy admin or generator modules are being loaded into the same server process as the public site
- identify whether request handlers perform expensive synchronous work
- identify whether pages load full datasets when only summaries/previews are needed
- detect whether server-side rendering or page bootstrap is loading too much data

C. FRONTEND / BUNDLE / ROUTE PERFORMANCE
- audit the frontend bundle
- detect oversized chunks
- detect large shared chunks affecting all routes
- find routes importing admin-only or builder-only libraries into customer-facing pages
- identify heavy component trees
- identify eager loading of translations/locales
- identify giant JSON imports
- identify duplicated libraries
- identify pages that block rendering on large network requests
- find hydration bottlenecks, rendering loops, unnecessary global providers, and repeated client-side fetch storms

D. DATABASE / QUERY / CONTENT ACCESS
- audit all high-traffic queries
- identify N+1 queries
- identify missing indexes
- identify queries returning too many columns or rows
- identify places where entire lessons/questions/rationales/translations are loaded when only metadata is needed
- identify slow joins across questions, lessons, subscriptions, user progress, multilingual content, or exam datasets
- identify whether queries are tier-unsafe and loading cross-tier content unnecessarily
- inspect connection pooling, timeouts, retries, and failure handling
- inspect whether pages are blocked by sequential DB calls that should be parallelized or cached

E. CONTENT / GENERATOR / ASSET SYSTEMS
- inspect whether AI generation, PDF generation, image generation, sitemap generation, lesson/question transforms, or translation preparation are occurring in the live request path
- inspect whether admin generators are bundled into customer-facing runtime paths
- identify huge content blobs or long-form rationale payloads being transferred where they are not needed
- identify any place where generated content is stored or loaded inefficiently
- identify whether content health audits or enrichment jobs are running too often or at the wrong time

F. TRANSLATION / I18N SYSTEM
- inspect whether all language packs are loaded at startup
- inspect whether all locales are bundled into the initial payload
- inspect whether language resolution causes redundant requests
- detect whether translated page variants are generating route explosion or build bloat
- inspect whether fallback logic is too heavy
- identify hardcoded strings or translation loading patterns that increase bundle size or memory

G. DEPLOYMENT ARCHITECTURE
- inspect whether the application is too monolithic for the current deployment model
- determine whether public frontend, backend API, admin system, content generators, and scheduled jobs should be separated
- identify what should be static, what should be dynamic, and what should be background-only
- identify whether scaling configuration is insufficient
- identify whether memory allocation is too low for current architecture
- identify whether one deployment is doing work that should be split into multiple services or execution modes

SECTION 3 — MANDATORY FIXES TO IMPLEMENT
==================================================

After the audit, implement the fixes safely.

A. BUILD / PUBLISH OPTIMIZATION
Implement all reasonable improvements such as:
- remove all non-essential tasks from publish/build flow
- ensure publish only installs, builds, and starts
- move migrations/backfills/repair scripts out of standard deploy path unless explicitly required
- remove generation jobs from publish
- reduce build context size
- exclude unnecessary generated artifacts
- optimize asset handling
- split bloated route generation where possible
- ensure source maps/debug outputs are appropriate for production
- improve startup speed
- document exact before/after deploy behavior

B. MEMORY STABILIZATION
Implement:
- removal of unbounded in-memory caches
- memory-safe pagination everywhere needed
- reduction of oversized payload creation
- elimination of request-time heavy transforms
- cleanup of large startup loads
- lazy initialization of expensive modules
- prevention of admin/generator logic loading into public runtime unless needed
- better request timeouts and guards
- memory-conscious serialization/deserialization

C. FRONTEND OPTIMIZATION
Implement:
- route-level code splitting
- lazy loading for admin-only, builder-only, analytics-heavy, chart-heavy, editor-heavy, PDF-heavy, and generator-heavy modules
- load locale files on demand
- reduce initial bundle size
- reduce shared chunk pollution
- optimize data fetching to prevent overfetch
- prevent repeated client-side requests
- add sensible loading boundaries and route error boundaries
- ensure public routes are not importing internal admin dependencies unless required

D. DATABASE / API OPTIMIZATION
Implement:
- pagination where missing
- narrower SELECTs
- correct indexes for high-frequency filters and joins
- removal of N+1 query patterns
- caching for high-traffic read paths where safe
- request coalescing or batching where useful
- timeouts and graceful degradation
- only loading entitlement-appropriate data
- preventing cross-tier overfetching

E. BACKGROUND JOB SEPARATION
Move out of request-time execution anything heavy such as:
- AI generation
- PDF generation
- image generation
- large content transforms
- sitemap rebuilds
- translation batch work
- bulk publishing/reindexing
- health audits
These must not block live customer traffic.

F. FALLBACKS / RESILIENCE
Implement:
- route-level fallbacks
- partial render capability
- cached last-known-good responses where appropriate
- graceful handling when one subsystem fails
- no dead-end page for paid users
- safe default states
- timeout fallback UIs
- isolated failure boundaries so one feature failure does not break the whole page

G. LOGGING / MONITORING / ALERTING
Add or improve:
- structured error logging
- memory and slow-route instrumentation
- slow-query logging
- build timing visibility
- route performance diagnostics
- publish diagnostic output
- warnings for oversized payloads or chunk regressions
