ENTITLEMENTS & LIMIT LOGIC (NON-NEGOTIABLE)
Implement entitlement checks on the server for every mutation:
- When creating/updating a card:
  - Determine user entitlement:

PAYMENTS IMPLEMENTATION (DEFAULT: STRIPE CHECKOUT + WEBHOOKS)
1) Server route: POST /api/billing/deck-upgrade/create-checkout
   - auth required
   - input: deckId
   - verify: user owns deck; deck not already upgraded
   - create Stripe Checkout Session for one-time payment
   - include metadata: userId, deckId, purchaseType='deck_upgrade'
   - return checkout URL
2) Webhook route: POST /api/billing/stripe-webhook
   - verify Stripe signature
   - on checkout.session.completed or payment_intent.succeeded:
     - validate metadata
     - mark purchase paid
     - set deck.isUpgraded=true, deck.upgradedLimit=300, deck.upgradedAt=now
     - write AuditLog entry (if present)
3) Handle refunds:
   - if refunded, set isUpgraded=false OR freeze additional cards (see policy below)
   - Default policy: On refund, lock deck to free rules; if deck exceeds free cap, block adding new cards and prompt user to delete cards or re-upgrade.
