You are working on the live NurseNest production system.

We have a CRITICAL operational reliability issue.

Current behavior is unacceptable for a growing subscriber platform:
- the system sends repeated alert emails every few seconds/minutes
- one underlying incident triggers multiple duplicate notifications
- memory failures and emergency mode loops are causing alert storms
- the system is not safely handling instability
- this will become worse as subscriber count and concurrent usage increase

Your task is to implement a PERMANENT SUBSCRIBER-SAFE SCALING AND ALERT-CONTROL FIX.

The goal is to ensure that as traffic, subscribers, and paid users grow:
- the app remains stable
- alerts remain actionable
- operators are not spammed
- users are protected from outages
- one incident does not become platform chaos

==================================================
SECTION 1 — INCIDENT GROUPING AND ALERT FLOOD PREVENTION
==================================================

Redesign the alerting system so repeated failures do NOT send endless duplicate emails.

Implement:
- incident grouping by root cause
- deduplication windows
- per-alert cooldowns
- suppression of duplicate notifications within the same incident window
- one rolling incident record for repeated memory/emergency/service-down failures
- operator summaries instead of repeated individual alerts

Each incident should track:
- incident_id
- root cause category
- severity
- first_seen
- last_seen
- occurrence_count
- affected components
- current status (open, monitoring, resolved)

Emails should only send when:
- a new incident opens
- severity materially worsens
- incident remains unresolved after a threshold
- incident resolves
- manual escalation is needed

Do NOT send a new email every few seconds for the same ongoing problem.

==================================================
SECTION 2 — ALERT TAXONOMY AND TEMPLATE CORRECTNESS
==================================================

Fix alert classification completely.

Current issue:
- reliability incidents are mislabeled as “Payment Failed”
- different operational issues are being mixed together confusingly

Create clear alert categories:
- memory_critical
- emergency_mode_activated
- service_down
- synthetic_test_failure
- content_integrity_failure
- deploy_freeze
- payment_failure (only for actual payment failures)
- reliability_warning

Ensure:
- title matches category
- body matches category
- operator can tell what actually failed immediately
- no misleading labels remain

==================================================
SECTION 3 — LOAD-SAFE SUBSCRIBER GROWTH ARCHITECTURE
==================================================

Assume the platform will gain more subscribers and concurrent traffic.

Audit whether the system can safely scale with more users.

Inspect:
- exam loading under concurrency
- memory growth per request
- concurrent heavy endpoint behavior
- whether heavy jobs run in the web process
- whether request pileups increase memory
- whether retries amplify load
- whether emergency mode actually reduces demand

Implement safeguards so subscriber growth does NOT cause alert floods or outages:
- concurrency limits on heavy routes
- queueing or rejection for expensive jobs
- load shedding when memory or latency is high
- lightweight fallbacks for overloaded routes
- autoscaling or right-sized baseline resources
- worker separation for heavy jobs

==================================================
SECTION 4 — CORE USER FLOW PROTECTION
==================================================

Protect critical paid-user flows under load.

Prioritize:
- login
- dashboard
- exam list
- exam open flow
- question answering
- subscription/account access

If the system is degraded:
- preserve core flows first
- disable nonessential features first
- provide lightweight fallbacks
- never return dead-end pages where possible

The system must be designed so:
- paid users do not lose access because optional heavy features consume resources
- traffic spikes do not collapse the whole app

==================================================
SECTION 5 — MEMORY AND TRAFFIC CONTAINMENT
==================================================

Prevent traffic growth from turning into repeated memory incidents.

Implement:
- bounded caches
- request size limits
- response size limits
- exam chunking/pagination
- rationale lazy loading
- max concurrent exam opens
- max concurrent heavy jobs
- route-specific timeouts
- memory threshold load shedding
- emergency mode actions that actually reduce load

Emergency mode must:
- disable or degrade nonessential expensive routes
- preserve core user traffic
- stop repeated self-destruction loops
- not just send more alerts

==================================================
SECTION 6 — POST-DEPLOY AND LIVE TRAFFIC SAFETY
==================================================

Add safeguards so broken changes are caught before and after release.

Implement:
- post-deploy smoke tests for critical flows
- alert flood detection
- incident-rate anomaly detection
- deploy freeze only when necessary and with proper operator summary
- rollback recommendation or auto-rollback criteria if critical flows fail after deploy
