You are a senior full-stack engineer building a secure Admin Authoring System (CMS-like) inside my nursing education platform called NurseNest.

GOAL:
Create an admin-only area where I (the site owner) can:
- Create / edit / publish lessons
- Create / edit / publish questions for the question bank and practice exams
- Create / edit / publish OSCE simulations (JSON-driven)
- Upload and manage images (for lesson graphics and OSCE scenes)
- Organize content by tier (RPN / RN / NP), category, topic, difficulty
- Preview content before publishing
- Version content (draft vs published)

CONTEXT:
This is a custom platform (not using Webflow/Wix). It is built with Next.js (App Router), TypeScript, Tailwind. Database can be PostgreSQL (preferred) using Prisma. If the repo already uses another DB, adapt to it.

SECURITY REQUIREMENTS (CRITICAL):
- Admin routes must be protected by authentication (e.g., NextAuth or existing auth)
- Admin access must be role-based (admin role in DB)
- Block access server-side (not just client-side hiding)
- Add audit logging: who changed what and when
- Validate inputs, prevent XSS (sanitize HTML if rich text used)
- Rate limit admin endpoints minimally to prevent abuse

CONTENT MODEL REQUIREMENTS:

A) LESSONS
Lesson fields:
- id, title, slug, tier (RPN/RN/NP), category, topic, subtopic
- difficulty (beginner/easy/moderate/hard)
- learning objectives
- body content (rich text or markdown)
- clinical pearls
- red flags
- scope-of-practice notes (RPN/RN/NP)
- references (optional)
- status: draft/published
- updated_at, created_at, published_at
- SEO: meta_title, meta_description, canonical_slug
- Related lessons (many-to-many)
- “Key lab values” block (optional structured data)

Editing UX:
- A list view with filters/search
- A lesson editor with autosave
- Preview mode
- Publish/unpublish button
- Duplicate lesson button

B) QUESTION BANK
Question fields:
- id, tier, topic, difficulty, type (SBA/SATA/ordered/case/medmath)
- stem, options, correct answer(s)
- correct rationale
- rationales for why wrong options are wrong
- key teaching points
- exam trap note
- tags, clinical setting
- status: draft/published
- version history

Editing UX:
- Question list with filters
- Question editor with validation (must have correct answers, no duplicate options)
- Bulk import/export JSON
- Publish/unpublish

C) OSCE SIMULATIONS
Simulation fields:
- id, title, tier, category, difficulty
- scene image asset
- JSON definition for steps/hotspots/scoring
- status: draft/published
- preview mode that runs the simulation in admin preview
- import/export JSON

D) MEDIA / ASSETS
- Upload images (store locally or in S3-compatible storage)
- Tag images (e.g., “OSCE scenes”, “Lesson graphics”)
- Insert image into lesson body
- Keep stable URLs for published assets

E) AUDIT LOG
Track:
- admin user id
- content type (lesson/question/simulation)
- content id
- action (create/edit/publish/delete)
- timestamp
- diff summary if possible

DELIVERABLES:
1) Prisma schema (or SQL schema) for all content types + admin role + audit log
2) Next.js admin route structure:
   - /admin (dashboard)
   - /admin/lessons
   - /admin/questions
   - /admin/simulations
   - /admin/media
   - /admin/audit
3) Protected route middleware (server-side) enforcing admin role
4) Editor UIs built with clean components (Tailwind), mobile-friendly
5) API routes or server actions for CRUD with validation and sanitization
6) JSON import/export utilities for questions and simulations
7) Seed script to create the first admin user and sample content
8) Short README explaining how I add content as an admin and publish

IMPORTANT IMPLEMENTATION RULES:
- Do not hand-wave: provide copy/paste-ready code with file paths.
- Use robust validation (zod preferred).
- Rich text can be markdown editor or a safe WYSIWYG; if WYSIWYG, sanitize HTML.
- Ensure published content is what the public pages render.
- Draft content must not leak publicly.

OUTPUT FORMAT:
- Start with a brief architecture plan
- Then show Prisma schema
- Then show file-by-file code blocks for key pages, components, and route protection
- Then provide setup steps (migrate DB, seed admin, run dev)

If any existing project structure already exists, adapt to it instead of replacing it.
Ask for existing file paths only if absolutely necessary; otherwise make reasonable assumptions.
