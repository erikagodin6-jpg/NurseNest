Upgrade all NurseNest AI content generators AND implement a strict hardcoded string scanner so the platform enforces full multilingual correctness at generation, storage, runtime, and build levels.

This is a **hard enforcement system**. No partial compliance is acceptable.

---

## GLOBAL ENFORCEMENT GUARANTEE

The system MUST guarantee:

→ No incorrect-language content can be generated
→ No invalid content can be saved
→ No invalid content can be published
→ No mixed-language content can render
→ No hardcoded user-facing strings can ship

If any rule is violated:
→ the system must BLOCK execution (not warn)

---

## CRITICAL FAILURE BEHAVIOR

If validation fails at ANY stage:

* Generation → abort + retry or fail
* Validation → reject payload
* Storage → block write
* Publishing → block publish
* Rendering → show controlled fallback UI
* Build → fail build

No silent failures. No soft warnings.

---

## SECTION 1 — AI GENERATION ENFORCEMENT (STRICT)

Applies to ALL generators:

* question generator
* rationale generator
* lesson generator
* flashcard generator
* CAT exam generator
* blog/content generator
* allied health generators
* new grad generators

---

1. HARD INPUT CONTRACT (NON-BYPASSABLE)

---

Each generation request MUST include:

* target_language (required, ISO code)
* locale (optional but supported)
* content_type
* exam/tier/profession (if applicable)

System MUST:

* validate presence before execution
* throw blocking error if missing

---

2. LANGUAGE-LOCKED PROMPT SYSTEM (CENTRALIZED)

---

* All generators must use a shared base prompt template
* No generator may define its own language behavior independently

Prompt MUST enforce:

* 100% output in target_language
* zero mixed-language tolerance
* no English unless explicitly requested
* preserve “NurseNest” only
* enforce clinical + exam-specific terminology

---

3. FIELD-LEVEL HARD VALIDATION

---

Validate EACH field independently:

* titles
* stems
* scenarios
* answer choices
* rationales (correct + incorrect)
* explanations
* clinical pearls
* hints
* flashcards
* lesson sections
* metadata labels

If ANY field fails:
→ reject ENTIRE object

---

4. LANGUAGE DETECTION (MANDATORY + REDUNDANT)

---

* run detection per field
* run detection on full payload
* compare against requested language

If mismatch:
→ reject
→ retry OR fail

---

5. TERMINOLOGY ENFORCEMENT (STRICT MODE)

---

* implement controlled terminology dictionary
* enforce:

  * consistency
  * correctness
  * exam alignment

Violations:
→ flag OR reject depending on severity

---

6. RETRY SYSTEM (CONTROLLED)

---

* max 2 retries
* each retry must:

  * increase strictness
  * reinforce language constraint

If still failing:
→ status = validation_failed
→ route to admin queue
→ DO NOT SAVE OR PUBLISH

---

7. ASYNC JOB SAFETY (CRITICAL)

---

For background/queued generation:

* validation MUST run before job completion
* jobs MUST NOT write partial/invalid content
* failed jobs must:

  * mark status
  * log reason
  * not retry infinitely

---

8. STRUCTURED OUTPUT (STRICT CONTRACT)

---

Every generator MUST return:

{
requested_language,
detected_language,
field_validation: {
title: boolean,
stem: boolean,
rationales: boolean,
...
},
validation_passed,
terminology_check_passed,
retry_count,
status: "approved" | "validation_failed" | "needs_review"
}

---

9. STORAGE ENFORCEMENT (TRANSACTIONAL)

---

* wrap save in transaction

* if ANY validation fails:
  → rollback

* save ONLY:

  * correct language
  * validated content
  * proper translation row

---

10. CACHE + CDN SAFETY (CRITICAL)

---

* cache must be language-scoped
* NEVER serve cached content from another language
* invalidate cache on:

  * language change
  * content update
  * translation update

---

11. PUBLISHING GATE (HARD BLOCK)

---

Content MUST NOT publish unless:

* validation_passed = true
* terminology_check_passed = true
* translation_status = approved
* NOT stale
* matches requested language

---

## SECTION 2 — HARDCODED STRING SCANNER (ZERO TOLERANCE)

---

1. AST-BASED DETECTION (REQUIRED)

---

* must use AST parsing (Babel/TS parser)
* NOT regex/grep

---

2. VIOLATION SEVERITY TIERS

---

Define:

* CRITICAL:
  visible UI text (buttons, headings, CTAs)
* HIGH:
  cards, labels, tooltips
* MEDIUM:
  metadata, secondary UI
* LOW:
  logs (optional)

---

3. BUILD-BLOCKING RULES

---

Build MUST FAIL if:

* ANY critical violations exist
* total violations exceed threshold

---

4. AUTO-FIX SUPPORT (OPTIONAL BUT RECOMMENDED)

---

Suggest:

* wrapping with t('...')
* extracting to locale file

---

5. CONTINUOUS ENFORCEMENT

---

Run scanner in:

* pre-commit hook
* CI pipeline
* production build

---

## SECTION 3 — RUNTIME SAFETY

---

1. LANGUAGE ISOLATION

---

* every request scoped by language
* no cross-language data merging

---

2. FALLBACK UI (CONTROLLED)

---

If content invalid/missing:

→ show structured message (localized)
→ DO NOT fallback to English silently

---

3. MIXED-LANGUAGE DETECTION

---

* detect at render layer
* if detected:
  → block render OR replace with fallback
