1. Keep the service working under load

==================================================
SECTION 1 — ARCHITECTURE SEPARATION
==================================================

Audit the current architecture and identify what heavy workloads are running inside the live web process.

Inspect whether the web app is also handling:
- AI generation
- bulk question generation
- exam assembly
- PDF/image generation
- scheduled audits
- content validation jobs
- repair/regeneration jobs
- analytics aggregation
- other long-running or memory-intensive tasks

Implement a production-safe separation:
- web process handles lightweight request/response traffic only
- heavy or long-running work moves to workers/queues/background jobs

The live user-facing service must remain lightweight and reliable.

SECTION 2 — EXAM SYSTEM HARDENING
==================================================

Trace the full exam_open_flow and all exam-related endpoints.

Implement permanent memory-safe and reliability-safe behavior:
- load exam shell/metadata first
- load questions in chunks/pages/sections
- load rationales and heavy metadata only when needed
- avoid giant all-at-once payloads
- store attempt state in durable storage, not in-process memory
- cap concurrent exam loads
- add route timeouts and circuit breakers
- add graceful fallback if a full exam cannot load

Requirements:
- no dead-end exam screen
- no blank page
- one failing exam must not affect the whole platform
- exam routes must not be able to crash the service

==================================================
SECTION 3 — MEMORY AND LOAD CONTAINMENT
==================================================

Perform a full audit of memory pressure, leaks, and load amplification.

Inspect and fix:
- oversized DB queries
- oversized API responses
- unbounded caches
- global arrays/maps/objects growing over time
- duplicate object expansion/serialization
- retry loops
- repeated listeners/timers
- frontend refetch loops
- concurrent heavy route pileups
- background jobs sharing the web process
- synthetic tests worsening instability

Implement:
- bounded caches with eviction
- response trimming
- request size limits
- response size limits
- concurrency limits on heavy endpoints
- queueing or rejection for expensive work
- route-level memory logging
- load shedding at warning/critical thresholds

When memory or latency is high:
- preserve core user flows
- degrade or disable nonessential expensive features
- do not continue attempting heavy work while the service is unstable

- show exact validation reasons in admin
- add safe repair/regeneration tooling
