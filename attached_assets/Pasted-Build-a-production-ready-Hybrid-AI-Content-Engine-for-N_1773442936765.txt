Build a production-ready Hybrid AI Content Engine for NurseNest that minimizes per-generation API costs by using a layered system instead of relying on one premium model for everything.

PROJECT GOAL
Create an internal admin-only content generation engine for NurseNest that can generate:
1. Exam-style questions
2. Flashcards
3. Lesson drafts
4. Rationales
5. Clinical pearls
6. SEO blog/article drafts
7. Image matching suggestions from already-uploaded internal images

The engine must prioritize:
- medical accuracy
- structured educational quality
- duplicate prevention
- tier correctness
- low operating cost
- human review before publishing

CRITICAL BUSINESS RULES
- Never publish generated content directly to production without passing validation and QA flags
- Never write generated content only to dev if the admin explicitly selects production mode
- Add a clear environment toggle: DEVELOPMENT / STAGING / PRODUCTION
- Default admin generation target must be the published database only when the admin explicitly selects PRODUCTION
- Log every generation job, validation result, review state, and publish action
- All generation must be reproducible via saved prompts, metadata, model used, and timestamps
- Do not overwrite existing published content unless explicitly approved by admin
- Never mix tiers incorrectly
- Never show RPN content inside RN or NP learner-facing pages
- Never show NP content inside RPN or RN learner-facing pages
- Never show tier labels like “RPN”, “RN”, or “NP” inside the lesson title shown to the learner unless the admin specifically wants that
- Maintain NurseNest visual and structural consistency

HIGH-LEVEL ARCHITECTURE
Build a hybrid engine with the following layers:

LAYER 1: CONTENT BLUEPRINT ENGINE
- Use structured templates and topic metadata first
- The system should assemble a generation blueprint before any model call
- Blueprints must include:
  - topic
  - slug
  - tier
  - exam type
  - locale (Canada, US, both)
  - body system
  - specialty
  - difficulty
  - client needs category
  - lesson category
  - learning objectives
  - required sections
  - prohibited phrases
  - image tags
  - SEO keywords where applicable

LAYER 2: LOW-COST DRAFT GENERATION
- Support a local/open-source model pathway for first drafts
- Design provider abstraction so the engine can swap between:
  - local model
  - Ollama
  - vLLM
  - LM Studio
  - paid fallback API
- Create a provider interface so the app can switch models without changing business logic
- Use the cheapest/local model for:
  - first-pass lesson drafts
  - question stems
  - flashcard fronts/backs
  - clinical pearl drafts
  - metadata enrichment

LAYER 3: RULES ENGINE
- After draft generation, run a strict rules engine that enforces:
  - required headings
  - title cleanup
  - no learner-facing tier names in titles
  - length requirements
  - formatting consistency
  - rationale structure
  - answer option count
  - no empty sections
  - no duplicate answer choices
  - no contradictory statements
  - no casual or unsafe wording
  - no cartoonish or non-academic phrasing
- Strip phrases like:
  - “As an AI”
  - “consult your doctor”
  - “this may vary”
  - any filler language that weakens exam prep usefulness
- Ensure all lesson titles are concise and professional

LAYER 4: MEDICAL VALIDATION + SAFETY CHECKS
- Build a validation pipeline that checks generated nursing content before review
- The validator must check:
  1. structure validity
  2. medical terminology sanity checks
  3. internal contradiction detection
  4. answer key consistency
  5. rationale supports the correct answer
  6. distractors are plausible but clearly incorrect
  7. Canadian vs US terminology conflicts
  8. tier appropriateness
  9. forbidden claims or unsafe statements
  10. suspected hallucinations
- Add a “confidence score” and “needs clinical review” flag
- Add automatic hard-fail rules for:
  - wrong answer/rationale mismatch
  - pediatric/adult dosing claims without verified source fields
  - fabricated citations
  - missing rationale
  - duplicate slug/title conflict
  - severe tier mismatch
  - unsupported medical numeric claims without source support
- If validation fails, save as DRAFT_REJECTED or NEEDS_REVISION and do not publish

LAYER 5: SIMILARITY + DUPLICATE DETECTION
- Check all newly generated content against existing published and draft content
- Use embeddings or text similarity checks for:
  - near-duplicate question stems
  - near-duplicate rationales
  - repeated lesson intros
  - repeated flashcard phrasing
- Add configurable thresholds:
  - low similarity warning
  - moderate similarity review
  - high similarity block
- If similarity exceeds the block threshold, do not publish

LAYER 6: HUMAN REVIEW GATE
- Build admin workflow states:
  - draft
  - generated
  - validated
  - needs_revision
  - approved
  - published
  - rejected
- Only APPROVED content can be published
- Add reviewer notes and clinical editor notes
- Add a one-click compare view for:
  - generated text
  - validator findings
  - existing similar content
  - final edited version

CONTENT TYPES TO SUPPORT

1. EXAM QUESTIONS
For exam questions, generate:
- question stem
- 4 answer choices unless CAT-specific logic requires alternative format
- correct answer
- rationale for correct answer
- brief explanation for incorrect answers except where CAT exam rules exclude this
- clinical pearl
- difficulty
- topic tags
- blueprint mapping
- locale
- tier
- image suggestion tag if appropriate
- optional linked lesson slug

Question rules:
- Must sound like premium nursing exam-prep content
- Must be clinically plausible
- Must not be trivial
- Must not be misleading
- Must match tier level:
  - RPN/LVN = foundational practical nursing
  - RN/NCLEX = broader acute care and higher clinical judgment
  - NP = advanced diagnosis, management, prescribing context where appropriate
- Avoid overlong stems unless clinically necessary
- Rationales must be educational, concise, and high-yield

ACCURACY STRATEGY
This is critical:
Do not pretend AI-generated medical content is inherently accurate.
Build the system to maximize accuracy using the following:

1. Source-supported fields
- Add database columns for source references
- Allow admin to attach:
  - textbook source
  - guideline source
  - manual source
  - instructor-edited source
- If the engine makes numeric clinical claims, medication details, diagnostic criteria, or guideline-level statements, mark content as REQUIRES_SOURCE unless source is attached

2. No fabricated citations
- Never invent references
- If no verified source exists, leave references blank and flag for review

3. Medical review flags
- Add flags like:
  - requires_clinical_review
  - requires_source_review
  - possible_hallucination
  - locale_conflict
  - answer_rationale_mismatch

4. Safe scope control
- For medication dosing, prescribing, disease staging, or fast-changing guideline content:
  - either avoid generating precise claims by default
  - or require manual source confirmation before approval
