CRITICAL LONG-RUNNING BATCH JOB RULE FOR NURSENEST

The AI generation system must support long-running batch jobs for questions, CAT/adaptive pools, flashcards, lessons, and related content without depending on a browser session staying open and without failing just because generation takes a long time.

Do NOT implement generation as a single synchronous request that times out.
Do NOT depend on the admin page remaining open.
Do NOT lose progress if one batch fails midway.

Instead, implement a durable background job architecture with resumable workers and concurrent batch execution.

====================================================
BACKGROUND JOB ARCHITECTURE
====================================================

All large generation tasks must run through a queue/worker system.

Examples:
- generate 500 questions
- generate 5 specialty batches at once
- generate 300 flashcards
- generate full topic packs across multiple specialties
- generate allied health page content in parallel
- run bulk SEO/content cleanup jobs

Use a durable jobs table and worker-based processing model.

Each batch generation request must create:
- one parent job
- multiple child batch jobs/items

Example:
Generate 500 questions for Emergency Nursing
→ parent job
→ 10 child batches of 50 each

====================================================
NO FRONTEND TIMEOUT DEPENDENCY
====================================================

The system must:
- continue processing if the admin closes the browser
- continue processing if the page is refreshed
- continue processing if the network disconnects
- allow the admin to return later and see progress

The frontend should only poll or subscribe to job status.
The actual generation must happen in backend workers.

====================================================
RESUMABLE PROCESSING
====================================================

If a worker crashes, process restarts, or one child batch fails:
- do not lose already completed items
- save progress incrementally
- mark failed child jobs separately
- allow retry from last incomplete step
- support resume of parent job without restarting everything

Each child batch should track:
- pending
- running
- completed
- failed
- retrying
- cancelled

Each generated item should be saved incrementally, not only at the end of the whole parent job.

====================================================
PARALLEL BATCH EXECUTION
====================================================

Allow multiple child batches to run at the same time.

However, do NOT allow unbounded concurrency.

Implement configurable concurrency controls:
- max concurrent parent jobs
- max concurrent child batches per parent job
- max concurrent model requests per provider
- max concurrent production writes
- separate concurrency caps for local model vs fallback provider

Admin must be able to configure concurrency in settings.

Example settings:
- maxParentJobs = 3
- maxChildBatchesPerJob = 4
- maxRequestsPerProvider = 6

====================================================
MULTI-BATCH SUPPORT
====================================================

The system must support:
- multiple independent batch jobs running at once
- multiple specialties running at once
- multiple page categories running at once
- multiple content types running at once

Examples:
- Emergency Nursing questions batch running
- Paramedic flashcards batch running
- New Grad prioritization batch running
- RN lesson expansion batch running

These should all be able to run concurrently within configured limits.

====================================================
WORKER SAFETY
====================================================

Workers must:
- claim jobs atomically
- avoid duplicate processing
- respect concurrency limits
- heartbeat while running
- mark stalled jobs for recovery
- retry transient failures
- stop retrying after configurable retry limit
- isolate failures so one batch does not kill the whole queue

====================================================
RATE LIMIT / RESOURCE PROTECTION
====================================================

Protect the system from overload.

Implement:
- provider-aware rate limiting
- queue backpressure
- memory/CPU safety checks where possible
- staggered retries with exponential backoff
- batch size controls
- provider cooldown on repeated failure

If the local model/provider is overloaded:
- pause or slow new child jobs
- do not crash the system
- do not silently drop jobs

====================================================
BATCH SIZE CONTROLS
====================================================

Add admin controls for:
- batch size
- concurrent batch count
- retry limit
- retry backoff
- provider timeout per request
- max runtime per child batch
- max runtime per parent job before manual review
- partial save frequency

Important:
Do not interpret “no timeout” as infinite hanging requests.
Instead, use:
- reasonable per-request timeout
- resumable retries
- durable background processing
- persistent partial saves

JOB STATUS UI
====================================================

Create a job monitor UI in AI Ops showing:
- parent jobs
- child batches
- queued/running/completed/failed counts
- percent complete
- provider used
- elapsed time
- estimated remaining batches
- retry count
- partial output counts
- validation status
- publish/review target
- cancel/pause/resume controls where safe

====================================================
CANCELLATION / PAUSE / RESUME
====================================================

Support:
- pause parent job
- resume parent job
- cancel parent job
- retry failed child batches only
- requeue incomplete child batches

Do not delete completed results when retrying failed parts.

NON-NEGOTIABLE RULES
====================================================

- Do not run large batch generation inline in the request/response cycle
- Do not require the admin page to stay open

- Do support pause, resume, retry, and recovery
