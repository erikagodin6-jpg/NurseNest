Refactor and stabilize the entire NurseNest pricing system so it is consistent, scalable, and resistant to rendering failures. The current system has issues including pricing cards not rendering, inconsistent architecture between nursing tiers and Allied Health, missing debug safeguards, and poor SEO due to a single pricing route.

This task should implement a unified pricing architecture across all tiers (RPN, RN, NP, Allied Health) with consistent plan structures, dedicated routes, and a shared rendering engine.

CORE OBJECTIVES

1. Fix the pricing card rendering issue for RPN, RN, and NP tiers.
2. Convert Allied Health pricing to match the same 4-card duration model used for nursing tiers.
3. Remove trial plans and any temporary test pricing plans.
4. Introduce dedicated SEO-friendly routes for each tier.
5. Ensure pricing cards are generated dynamically from a single pricing system rather than hardcoded components.
6. Add debugging safeguards so pricing cards cannot silently fail to render.
7. Ensure Stripe product mapping remains intact and consistent.

UNIFIED PRICING MODEL

All tiers must use the same duration-based pricing structure.

Duration tiers:
• 1 Month
• 3 Months
• 6 Months
• 12 Months

Example plan structure (CAD):

RPN:
• 1 Month — $24.99
• 3 Months — $59.99
• 6 Months — $99.99
• 12 Months — $149.99

RN:
• 1 Month — $29.99

• 12 Months — $159.99

NP:
• 1 Month — $34.99
• 3 Months — $79.99
• 6 Months — $119.99
• 12 Months — $169.99

Allied Health:
• 1 Month — $24.99
• 3 Months — $59.99
• 6 Months — $99.99
• 12 Months — $149.99

Remove all trial plans such as:
• 1 Day Trial
• 3 Day Trial
• Any temporary or sandbox pricing plans

ARCHITECTURE REFACTOR

1. Eliminate separate hardcoded pricing components such as allied-pricing.tsx.

2. Create one unified pricing engine that renders pricing cards dynamically based on:
   • tier key
   • plan duration
   • price data

All tiers must pull from the same pricing source.

Example tier keys:
• rpn
• rn
• np
• allied

Example data structure:

pricingPlans = {
rpn: [plans],
rn: [plans],
np: [plans],
allied: [plans]
}

The rendering component should read this data and generate cards automatically.

Do not duplicate pricing UI logic across multiple files.

ROUTING STRUCTURE

Create dedicated pricing routes:

/pricing
/pricing/rpn
/pricing/rn
/pricing/np
/pricing/allied

Behavior:

• /pricing should default to the RPN pricing tab
• /pricing/rpn loads the page with RPN plans active
• /pricing/rn loads the RN plans
• /pricing/np loads the NP plans
• /pricing/allied loads Allied Health plans

Important:
All routes must use the SAME pricing page component.

The route only determines which tier is selected.

Do NOT create separate pricing pages for each tier.

TAB BEHAVIOR

Keep the existing tab-style tier selector on the pricing page.

Tabs should update the URL automatically when selected.

Example:
Selecting RN updates the URL to:
/pricing/rn

This ensures SEO-friendly links while maintaining the current UI experience.

PRICING CARD RENDERING

Pricing cards must render dynamically based on the selected tier.

Each tier must show exactly four cards:

• 1 Month
• 3 Months
• 6 Months
• 12 Months

Cards should display:

• Plan duration
• Price in CAD
• Features included
• Call-to-action button
• Stripe price mapping

Mark the 6-month plan as “Most Popular”.

ERROR HANDLING AND DEBUG SAFEGUARDS

Implement a rendering guard to prevent silent failures.

If no plans are returned for a tier:

1. Log an error in the console.

2. Display a fallback message such as:
   “Pricing plans are temporarily unavailable. Please refresh or contact support.”

3. Do not render an empty pricing grid.

Also log:

• missing tier keys
• missing Stripe price IDs
• invalid plan objects

This ensures broken data does not silently break the UI.

STRIPE INTEGRATION

Ensure each pricing plan maps correctly to Stripe price IDs.

Requirements:

• Each duration plan must have a Stripe price ID.
• Stripe product mapping must remain tier-specific.
• Do not mix RPN/RN/NP/Allied price IDs.

The pricing component should load Stripe price IDs from configuration rather than hardcoding them in the UI.

Ensure checkout flows still work with Stripe payment links or checkout sessions.
