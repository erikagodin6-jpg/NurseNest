Act as a senior health-tech systems architect designing infrastructure for a professional nursing education platform called NurseNest.

Context:

NurseNest has two platform variants:
1) United States version
2) Canadian version

Each platform displays clinical guidance, best practices, and exam-relevant clinical frameworks.

Primary requirement:

Design a SAFE and DEFENSIBLE system that helps ensure clinical guidance remains current without allowing unverified AI-generated medical claims.

CRITICAL SAFETY CONSTRAINTS

- AI must NEVER autonomously modify clinical content
- AI must NOT invent or assume guideline updates
- All updates must originate from authoritative sources
- The system must support verification workflows
- The system must protect against outdated or incorrect medical guidance

SYSTEM OBJECTIVES

Create a conceptual design for a Guideline Monitoring & Update Assistance System with the following properties:

1) Source-Anchored Monitoring

- Track updates only from authoritative bodies

United States examples:
- CDC
- NIH
- FDA
- USPSTF
- Specialty societies (ACC, AHA, etc.)

Canada examples:
- Health Canada
- Public Health Agency of Canada (PHAC)
- Canadian Cardiovascular Society (CCS)
- CNA / specialty organizations

2) Change Detection Logic

- Detect when source documents change or update
- Identify version differences
- Flag potentially affected NurseNest content

3) Verification-First Workflow

- AI proposes POSSIBLE update areas
- Human review required before publication
- No automatic publishing allowed

4) Jurisdiction Separation

- Ensure Canadian site references Canadian guidance
- Ensure US site references US guidance
- Prevent cross-jurisdiction contamination

5) Auditability & Traceability

- Track when content was last reviewed
- Store source references
- Maintain revision history

6) Drift Prevention Mechanisms

- Alert when content ages beyond threshold
- Flag high-risk topics (drugs, protocols, dosing, emergencies)

7) AI Role Limitations

AI may:
- Summarize detected changes
- Compare document versions
- Identify impacted concepts

AI may NOT:
- Assert clinical correctness
- Generate new medical rules
- Replace human clinical oversight

OUTPUT REQUIREMENTS

Provide:

- System architecture overview
- Data flow model
- Update detection strategy
- Human verification workflow
- Risk mitigation safeguards
- Example lifecycle of a guideline update

Avoid:

- Claims of guaranteed accuracy
- Fully automated medical updating
- Unrealistic AI capabilities

3) Stripe safety patterns (chargeback + double-grant protection)
Non-negotiable Stripe rules
1.	Grant access only via webhook, not client-side success page.
2.	Verify webhook signature.
3.	Make processing idempotent (never grant twice for the same event).
Recommended Stripe flow (Replit-friendly)
•	Use Stripe Checkout for one-time products (Cram Pack, exam packs).
•	Use Stripe Subscriptions for monthly tiers.
•	Webhooks:
o	checkout.session.completed (for one-time purchases)
o	invoice.payment_succeeded (for subscriptions)
o	customer.subscription.updated / deleted
Idempotency pattern
Create a table stripe_events:
•	event_id (unique)
•	processed_at
•	type
Webhook logic:
•	If event_id already exists → return 200 OK (do nothing)
•	Else → process → insert event_id → commit
Entitlement record pattern (single source of truth)
entitlements table:
•	user_id
•	access_type (none | cram_pack | subscriber_rpn | subscriber_rn | subscriber_np)
•	access_expiry (nullable; for cram)
•	subscription_status (active/canceled/past_due)
•	created_at, updated_at
For exam credits:
exam_credits table:
•	user_id
•	product (mini/mock/intensive)
•	credits_remaining
Refund/chargeback handling (must exist)
•	If Stripe signals refund or dispute:
o	revoke cram pack entitlement (or set expiry to now)
o	freeze exam credits added by that purchase
This reduces disputes and abuse.
