Nest backend (Replit).

Implement a UNIFIED WEB-BASED BILLING SYSTEM for NurseNest so pricing stays identical across desktop and mobile app.

This is a pricing consistency and revenue protection task.

==================================================
PRIMARY OBJECTIVE
==================================================

Use the WEBSITE / STRIPE billing system as the single payment source of truth for NurseNest.

Requirements:
- same pricing on website and mobile app
- no billing inconsistency between platforms
- no duplicate pricing systems
- mobile app users can study on phone but subscribe through the same web checkout flow
- subscription state syncs back to both web and app after purchase

IMPORTANT:
Most users will primarily use the desktop version, but they must still be able to access their account and study on mobile after subscribing.

==================================================
SECTION 1 — BILLING STRATEGY
==================================================

Implement the website billing system as the canonical subscription flow.

Rules:
- Stripe web checkout is the main billing path
- app does NOT maintain a separate pricing system
- app does NOT invent different plans or plan names
- app must display the exact same plan names, prices, and billing intervals as the website
- backend is source of truth for pricing metadata shown in both clients

==================================================
SECTION 2 — CENTRAL PLAN CONFIGURATION
==================================================

Create or refactor a central billing plan config used by both web and app APIs.

Each plan should include:
- planId
- planName
- monthlyOrYearly label
- currency
- price
- interval
- description
- feature summary
- isFeatured
- sortOrder
- stripePriceId
- active status

Do NOT hardcode separate price values across multiple parts of the codebase.

Create a single source of truth that both:
- website checkout UI
- mobile app paywall API
can read from.

==================================================
SECTION 3 — PLAN CONSISTENCY
==================================================

Ensure the exact same plans shown on the website are returned to the mobile app.

Implement an endpoint such as:
GET /billing/plans

Response should include:
- exact active plans
- exact prices
- exact currencies
- exact labels
- featured/best-value flag if applicable

This endpoint should be used by:
- website pricing UI if helpful
- mobile app paywall UI
so there is no drift over time

==================================================
SECTION 4 — WEB CHECKOUT SESSION
==================================================

Implement:
POST /billing/create-checkout-session

Input:
- authenticated user
- selected planId
- optional returnUrl / successUrl / cancelUrl context
- optional source metadata (web or app)

Behavior:
- validate user
- validate planId against central plan config
- create Stripe Checkout session using the matching stripePriceId
- return checkout URL
- attach metadata for userId, planId, and purchase source

Purchase source metadata examples:
- web
- mobile_app

==================================================
SECTION 5 — APP PURCHASE FLOW SUPPORT
==================================================

The mobile app will use the SAME Stripe-backed website checkout flow.

Backend must support app-origin purchases by:
- creating checkout session normally
- returning secure checkout URL
- tagging purchase source as mobile_app when launched from app
- updating entitlements the same as any website purchase

Result:
- user sees same price
- pays through web checkout
- subscription updates centrally
- app reflects premium access after refresh

==================================================
SECTION 6 — STRIPE WEBHOOKS
==================================================

Implement or harden webhook handling for:
- checkout.session.completed
- invoice.paid
- customer.subscription.updated
- customer.subscription.deleted

Requirements:
- verify Stripe signature
- idempotent processing
- map events to correct user
- update central subscription record
- recalculate entitlements
- log all important transitions

Never grant premium only from frontend success.
Always grant access only after validated backend / webhook confirmation.

==================================================
SECTION 7 — SUBSCRIPTION / ENTITLEMENT UPDATE
==================================================

After successful purchase:
- set correct subscription record
- assign premium entitlements
- update lastVerifiedAt
- ensure GET /auth/me and GET /me/entitlements reflect the new access
- ensure both app and website can immediately read the new status after refresh

==================================================
SECTION 8 — RETURN / SUCCESS FLOW
==================================================

Support a clean post-checkout experience.

Implement:
- success redirect handling
- cancel redirect handling

For app-origin checkout:
- support a clean return path that lets the app refresh account state after purchase
- if deep linking is available, support app-safe redirect handling
- if not, return to a web confirmation page that instructs user to reopen/return to the app and refresh

==================================================
SECTION 9 — SAME PRICE EVERYWHERE ENFORCEMENT
==================================================

Enforce pricing consistency as a product rule.

Rules:
- app must not display price values hardcoded separately from backend plans
- website must not drift from plan config
- changing a plan price in backend updates all clients
- one canonical price per market/currency/plan

==================================================
SECTION 10 — ADMIN / SUPPORT VISIBILITY
==================================================

Add support visibility for:
- user current plan
- purchase source
- Stripe customer/subscription IDs if stored
- last successful payment sync
- entitlement status

This helps resolve cross-platform billing questions quickly.

==================================================
SECTION 11 — SECURITY / RESILIENCE
==================================================

Ensure:
- only authenticated users create checkout sessions
- planId is validated
- inactive plans cannot be purchased
- duplicate or delayed webhooks do not corrupt subscription state
- billing failures do not remove active access incorrectly
- recently verified paid users are protected by safe entitlement handling
