PART B — Clinical Verification + Safety Gate (Public + Admin)
Non-negotiable:
- AI should NOT directly publish content.
- Every deck/card must be checked before it can be posted publicly.
- If content is unsafe/incorrect/low-confidence, the user must be warned and blocked from posting unless they revise.

Implement a verification pipeline with these constraints:
1) Data model changes
- decks table: id, owner_user_id, title, description, status (draft/pending_review/published/rejected), created_at
- cards table: id, deck_id, front, back, tags, created_at
- ai_jobs table: id, user_id, job_type (generate_deck/add_cards/rewrite/verify), input_json, output_json, status, created_at
- verification_reports table:
  - id, entity_type (deck/card), entity_id
  - verdict (pass / pass_with_edits / fail / needs_human_review)
  - issues_json (array of issues with severity)
  - citations_json (sources used)
  - confidence_score (0-1)
  - checked_at, model_version

2) Verification logic
Create a server-side “VerifyClinicalContent” function that:
- Checks for:
  - unsafe dosing, contraindications, wrong lab ranges, wrong triage priority, outdated guideline claims
  - hallucinated facts, made-up drugs, incorrect definitions
  - scope-of-practice errors (UAP/LPN/RN/NP)
- Returns:
  - verdict + confidence
  - list of issues with categories: “incorrect”, “unsafe”, “unclear”, “needs_source”, “scope”
  - required edits (if pass_with_edits)
  - citations list (guidelines/textbook-level sources)

Important:
- “Verified” must mean: passed checks with citations and acceptable confidence.
- If confidence < threshold OR issues include “unsafe/incorrect”, block publishing and show a warning.

3) User experience (gating)
- When user tries to create or edit a deck:
  - Run verification automatically on save (server-side).
  - If FAIL: show a modal warning:
    “This content appears medically incorrect or unsafe. Please revise before posting.”
    Provide issue list and highlight problematic cards.
    Disable “Publish” button.
  - If PASS_WITH_EDITS: show required edits checklist and allow publishing only after edits applied.
  - If NEEDS_HUMAN_REVIEW: allow saving as draft but block publish unless admin approves.

4) Admin override
- Admin dashboard page to review flagged decks and approve/reject.
- Audit log for any override.
