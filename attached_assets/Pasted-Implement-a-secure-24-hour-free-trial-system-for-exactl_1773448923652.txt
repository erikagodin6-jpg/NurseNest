Implement a secure 24-hour free trial system for exactly one subscription tier on the published NurseNest site.

Goal:
Allow a user to activate one single 24-hour free trial for one selected tier only, while reducing abuse and preventing easy content extraction. Do not rely only on IP. Build this for production database only, not development.

Core requirements:
1. Add a trial flow on pricing/signup for one selected tier.
2. Require account creation and verified email before trial activation.
3. Require a valid Stripe payment method before the trial starts, but do not charge until the trial ends unless canceled according to existing billing rules.
4. Only allow one free trial per user across the platform for that tier flow.
5. Prevent trial abuse using layered checks:
   - verified email uniqueness
   - Stripe customer/payment fingerprint if available
   - persistent device/browser token stored securely
   - IP address logging and rate limiting as a secondary fraud signal only
6. Store all entitlement and anti-abuse logic server-side.
7. Never trust client-side trial state.

Backend/data model:
1. Add trial fields to users/subscriptions tables or create a dedicated trial_entitlements table.
2. Track:
   - userId
   - selectedTier
   - trialStartedAt
   - trialEndsAt
   - trialStatus (active, expired, canceled, blocked)
   - verifiedEmailAt
   - stripeCustomerId
   - paymentFingerprint or equivalent non-sensitive dedupe field from Stripe metadata if available
   - deviceFingerprintHash
   - signupIp
   - lastSeenIp
   - abuseFlags JSON
3. Add server middleware to validate entitlement on every premium request.

Fraud controls:
1. Block new trial creation if:
   - same verified email already used a trial
   - same payment fingerprint already used a trial
   - same device fingerprint already used too many attempts
   - same IP created suspicious volume of trial attempts in a configurable time window
2. Add admin-visible abuse notes and trial audit logs.
3. Add rate limiting on signup, login, trial activation, and content fetch endpoints.

Content protection:
1. Disable PDF export, print export, and download/export buttons during trial.
2. Add subtle per-user watermark overlays on premium rationale panels, lesson images, and flashcard answer views using:
   - first name or masked email
   - user ID suffix
   - timestamp
3. Make watermark visually soft and premium, not ugly.
4. Reduce bulk scraping risk by limiting rapid sequential fetches and enforcing paginated authenticated API responses.
5. During trial, limit access to:
   - configurable max questions
   - configurable max flashcards
   - configurable max lessons
   - configurable mock exam preview only
6. Add backend counters for trial consumption and stop access when limits are reached even if 24 hours remain.

UI/UX:
1. Show a clean premium message:
   - “Start your 24-hour free trial”
   - “Card required. Cancel before renewal.”
2. Let the user choose one eligible tier only.
3. Show trial countdown in the dashboard.
4. Display a small note that content is view-only during trial and downloads/exports are unavailable.
5. Do not make the UI feel punitive or cheap.

Copy/paste protections:
1. For premium rationale and lesson text only, add light deterrents:
   - disable text selection on selected protected components where reasonable
   - block common copy shortcuts in protected views
   - block right-click in protected premium panels
2. Keep this limited to premium protected views only.
3. Do not break the rest of the site UX.
4. Add a feature flag to disable these controls easily if they hurt usability.

Accessibility and quality:
1. Do not globally disable keyboard access.
2. Do not interfere with navigation, forms, or screen-reader-friendly controls outside protected content panels.
3. Keep all protections modular and feature-flagged.

Admin dashboard:
1. Add trial analytics cards:
   - active trials
   - expired trials
   - converted trials
   - blocked abuse attempts
2. Add a trial audit table with filters by tier, date, IP, and status.
3. Add a fraud settings panel with configurable thresholds.

Technical requirements:
1. Production database only.
2. Use existing auth and Stripe patterns where possible.
3. Add clean TypeScript types, API routes, schema updates, and admin UI.
4. Add environment variables for fraud thresholds and trial limits.
5. Add clear comments explaining how trial validation works.
6. Do not break existing paid subscription flows.

Deliverables:
1. Database schema changes
2. Backend entitlement middleware
3. Trial signup and Stripe setup flow
4. Anti-abuse checks
5. Watermarked premium content components
6. Admin dashboard trial/fraud reporting
7. Configurable feature flags
8. Final summary of files changed and how to test in production safely
