You are a senior full-stack engineer building inside my existing NurseNest codebase. Implement a Quizlet-style flashcard system with user-generated decks, free-tier limits, learn/test study modes, sharing/privacy controls, and an AI “accuracy check” gate. You must reuse my existing design system and components so the flashcard pages match the NurseNest aesthetic exactly. Do not invent a new stack—inspect the repo and integrate with existing auth, database, storage, routing, and UI patterns.

ABSOLUTE REQUIREMENTS
1) Aesthetic parity: flashcard UI must look identical to the rest of the site (same nav, typography, spacing, rounded corners, pastel clinical tokens).
2) Security: all CRUD routes must enforce authentication; only owners can edit/delete their decks/cards; public decks are read-only to non-owners.
3) Free plan limits: free accounts can create decks but are limited to a TOTAL of 50 flashcards across all decks OR per deck (choose the one that matches the existing product model; default to 50 total across the account if no model exists). Enforce server-side.
4) Study modes:
   - Learn Mode: spaced retry of incorrect cards later in the session (and optionally across sessions).
   - Test Mode: no retry; user gets a final report card at end (score, time, weak topics).
5) AI accuracy check:
   - When a user creates/edits a flashcard, run an AI validation step that flags likely incorrect, unsafe, or misleading medical content.
   - Show a warning: “This may be inaccurate” + explanation + confidence.
   - User must choose: “Post anyway” (with visible disclaimer + flag marker) OR “Edit to fix”.
   - AI must never silently change the user’s text.
6) Sharing and privacy:
   - Deck visibility options: private (only owner), unlisted (anyone with link), public (discoverable/searchable).
   - Public decks: other users can save/duplicate to their library (optional but recommended).
7) Moderation & reporting:
   - Allow users to report a deck/card for inaccuracy/harmful content.
   - Admin can remove or unpublish user content from an admin panel (integrate with existing admin if present).
8) Performance:
   - Study mode should be snappy. Paginate large public deck browsing.
   - Avoid heavy client bundles; follow existing optimization patterns.

PHASE 4 — AI ACCURACY CHECK (GUARDRAILS)
Implement server endpoint: POST /api/flashcards/ai-check
Input: front, back, optional rationale, tier, tags
Output:
- status: pass/flag/unknown
- explanation: concise and helpful
- confidence
- suggestedCorrection (optional) BUT DO NOT auto-apply; show as suggestion only.

AI policy constraints:
- It must be an educational accuracy check only (no diagnosis or personalized medical advice).
- It must avoid hallucinated references. If providing references, require DOI/URL and mark unverified.
- It should be conservative: if uncertain, mark “unknown” rather than “pass”.

User experience:
- On save/publish, run AI check:
  - If pass: allow.
  - If flag: show warning modal with the explanation, highlight the suspect claim(s) if possible.
    User choices:
      1) “Edit” (return to editor)
      2) “Post anyway” → set userOverride=true and show a disclaimer badge on the card/deck like “Community card (not verified)”
- For public decks, if a card is flagged+override, show the disclaimer to all viewers.

IMPORTANT: If AI checking is expensive, implement:
- Debounced checks (only on explicit “Check accuracy” or on “Publish”)
- Caching by hash(front+back)
