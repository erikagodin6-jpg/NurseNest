SECTION 2 — MEMORY SAFETY SYSTEM (CRITICAL)
==================================================

Build a strict memory governance system across the app.

Requirements:
- Add process-level memory monitoring with periodic sampling of:
  - RSS
  - heap used
  - heap total
  - event loop lag
  - request concurrency
  - queue depth
- Define 3 memory states:
  - NORMAL
  - DEGRADED
  - CRITICAL
- Configure conservative thresholds well below process death.
- Enter DEGRADED before CRITICAL.
- Enter CRITICAL before OOM.
- Add hysteresis so the platform does not flap between states.

When DEGRADED:
- disable non-essential generators
- pause expensive background jobs
- reduce concurrency limits
- shorten payload sizes
- force exam endpoints into paginated/lightweight mode
- disable prefetching and cache warming
- suspend non-critical analytics/event fanout
- serve minimal responses where possible

When CRITICAL:
- block non-essential write-heavy or generation-heavy operations
- protect login, entitlements, checkout verification, existing content access, and active exam sessions first
- reject new heavy generation jobs with clear retry-safe responses
- isolate or terminate runaway workers
- shed optional traffic
- keep core user study flows alive in reduced mode

Also:
- identify and remove memory leaks
- eliminate unbounded arrays/maps/caches
- add TTLs and max sizes to in-memory caches
- ensure streams/buffers/files are properly closed
- prevent loading large exam/content datasets fully into memory
- replace bulk loading with pagination, chunking, lazy loading, and cursor-based retrieval
- ensure JSON serialization is bounded and safe
- add body size limits and output size limits
- add safeguards against accidental giant AI responses or giant content blobs
- add request timeouts and abort controllers for long-running operations
- ensure heavy processing is moved off the main request thread where possible

SECTION 3 — EXAM SYSTEM HARDENING (MISSION CRITICAL)
==================================================

The exam system is a protected revenue-critical flow and must never behave as an all-or-nothing feature.

Implement all of the following:

A. Exam retrieval safety
- Never fetch full exam banks when only a page/session slice is needed
- Serve questions in chunks/pages
- Only load the current question or a small window when possible
- Add lightweight exam session state storage
- Prevent giant nested payloads

B. Session resilience
- Preserve in-progress sessions
- autosave answers
- support restore-after-refresh/reconnect
- protect against partial backend failure
- if rationale or enrichment data fails, continue exam delivery without blocking the question

C. Fallback mode
If exam generation/open flow fails:
- offer fallback to cached published exam
- offer fallback to smaller exam set
- offer fallback to practice-by-category
- offer fallback to question-of-the-day style emergency mode
- never show a blank failure screen

D. Validation
- every question must have:
  - stable id
  - tier
  - exam type
  - question text
  - valid answer structure
  - minimum option count if MCQ type
  - correct answer
  - safe rationale handling rules
- reject malformed exam payloads before publish and before runtime delivery

E. Response shaping
- cap maximum questions returned per request
- cap rationale sizes
- strip unnecessary metadata from runtime exam payloads
- send only what the client needs for the current interaction

F. Synthetic monitoring
- create hardened synthetic tests for:
  - exam open flow
  - next question flow
  - submit answer flow
  - restore exam session flow
  - results flow
- these tests must run pre-deploy and post-deploy
- exam failures must block rollout

SECTION 4 — CONTENT INTEGRITY FIREWALL
==================================================

Build a strict content integrity firewall so invalid content cannot enter production.

Content types include:
- question banks
- exams
- flashcards
- lessons
- blog content
- scenarios
- allied health content
- multilingual content
- generated products
- preview/store content

Implement a 4-stage pipeline:

Stage 1: generation validation
- validate structure immediately after generation
- reject malformed output before save

Stage 2: business-rule validation
- tier correctness
- exam mapping correctness
- locale/language correctness
- required field completeness
- no empty lessons
- no zero-valid-item published bundles
- no mixed-language output
- no placeholder keys exposed to users
- no invalid pricing/store metadata
- no broken entitlement mappings

Stage 3: publish validation
- content cannot be marked published/store-ready/live unless all validations pass

Stage 4: runtime validation
- if corrupt content somehow exists, quarantine it automatically and do not serve it to end users

Rules:
- quarantine must be targeted, not catastrophic
- a spike in quarantined items must trigger investigation but should not collapse all content delivery
- published items with zero valid child items must be blocked at publish time
- empty lesson content must be blocked
- broken translations/i18n keys must be blocked from user-facing rendering
- all generators must produce schema-valid, language-valid, tier-valid output or fail safely

Also:
- add schema contracts for every content type
- add server-side validation, not just client-side
- add repair tooling/admin reporting for quarantined items
- add publish preflight summaries so admins can see exactly why a content item is blocked

SECTION 5 — PAYMENT, ENTITLEMENT, AND ACCESS PROTECTION
==================================================

Implement a central entitlement protection layer.

Requirements:
- one canonical access decision service
- access must be determined by plan, tier, region, product type, and subscription status
- RPN users cannot access RN/NP-only content
- RN users can access RN and allowed lower-tier content only
- NP users can access permitted NP/RN/RPN content
- Allied users can access only allied-appropriate content
- free users must see gated previews only where intended

Payment and subscription safety:
- payment failures must not be triggered merely because content payloads are malformed
- checkout and entitlement verification must be isolated from content generation issues
- add idempotency for payment webhooks and purchase processing
- add retry-safe reconciliation for billing desync
- preserve paid access during transient verification failures using short grace windows where appropriate
- never revoke valid paid access because of temporary backend instability

If entitlement lookup fails:
- prefer a safe temporary grace/fallback path for recently verified paid users
- log and flag for reconciliation
- do not hard lock legitimate users out in error

SECTION 10 — DEGRADED MODE / EMERGENCY MODE REDESIGN
==================================================

Emergency mode must become intelligent and layered.

Create modes:
1. NORMAL
2. GUARDED
3. DEGRADED
4. CRITICAL
5. RECOVERY

NORMAL
- all features available

GUARDED
- heightened monitoring
- lower concurrency on heavy paths
- prepare to shed optional load

DEGRADED
- pause generators
- pause bulk jobs
- protect exams, auth, entitlements, and access to already-published content
- serve smaller payloads
- disable non-essential enrichments

CRITICAL
- preserve only core routes
- protect active paying users
- block heavy admin/generation operations
- keep already-published study access available if safe

RECOVERY
- restore services gradually
- do not re-enable all heavy jobs at once
- confirm memory and error rate stabilization before exiting

Requirements:
- transitions must be automatic but stable
- every mode transition must be logged
- user-facing messaging must be calm and brief
- non-essential features must degrade gracefully, not explode
- emergency mode must not itself create excessive notifications
