1) Required data structures
A. Template blueprints (deterministic document plan)
Define these next to PRODUCT_PRESETS (as you planned):
type TemplateId = "cram" | "questions" | "cheatsheet" | "studyplan" | "bundle";
type ThemeId = "pastel" | "minimal" | "bold" | "neutral";

type SectionId =

  | "disorders"
  | "algorithms"
  | "questions"
  | "rationales"
  | "summary"
  | "bundle_overview"

type TemplateBlueprint = {
  id: TemplateId;
  label: string;
  minPages: number;
  maxPages: number;

  // budget model
  estCharsPerPage: number; // ex: 1400–2200 depending on typography
  sectionWeights: Partial<Record<SectionId, number>>; // must sum ~1.0 for main content

  // page types to generate in order (compiler uses this)
  pageFlow: Array<
    | { type: "cover" }
    | { type: "toc" }
    | { type: "divider"; section: SectionId; title: string }
    | { type: "section"; section: SectionId }
    | { type: "questions" }
    | { type: "rationales" }
    | { type: "summary" }
    | { type: "bundle" }
  >;

  // optional: repeated items for disorders, chapters etc.
  defaults?: {
    includeQuestions?: boolean;
    defaultQuestionCount?: number;
  };
};
You want pageFlow because it keeps generation deterministic and stops “AI decides layout.”

Step 1: Build plan + budgets
This is how you enforce “45 pages” in a predictable way.
function computeSectionBudgets(blueprint: TemplateBlueprint, targetPages: number) {
  const totalChars = targetPages * blueprint.estCharsPerPage;

  const weights = blueprint.sectionWeights;
  const budgets: Record<string, number> = {};

  let weightSum = 0;
  for (const w of Object.values(weights)) weightSum += (w ?? 0);

  for (const [section, w] of Object.entries(weights)) {
    budgets[section] = Math.round((totalChars * (w ?? 0)) / (weightSum || 1));
  }

  return { totalChars, budgets };
}
Step 2: AI output MUST be structured JSON
Do not let AI return prose blobs. Require JSON so you can render reliably.
Example schema:
type SectionContent = {
  title: string;
  blocks: Array<
    | { kind: "heading"; text: string; level: 1 | 2 | 3 }
    | { kind: "paragraph"; text: string }
    | { kind: "bullets"; items: string[] }
    | { kind: "table"; columns: string[]; rows: string[][] }
    | { kind: "callout"; title: string; body: string; flavor: "exam_tip" | "clinical_pearl" | "trap" }
  >;
};
Your AI call should say: “Return valid JSON only matching this schema.”
Step 3: Render pages using locked layouts
You should have page constructors:
renderCoverPage(theme, meta)
renderTOCPage(theme, toc)
renderDividerPage(theme, title)
renderSectionPages(theme, sectionContent, layoutVariants)
renderQuestionsPages(theme, questions)
renderRationalesPages(theme, rationales)
renderSummaryPage(theme, summary)
Each renderer returns your internal “elements” model.
Step 4: Auto-pagination + autofit (no overflow)
You already started this earlier (pagination + autofit). The compiler should run:
place elements in a flow container (known width/height minus margins)
if overflow, split remaining content into next page
for text blocks, autofit within bounds (shrink to min font or split)
Key rule: Never overflow in Guided Mode.
Step 5: Theme switching after compile
This is important for “different colour templates.”
Implement applyThemeToDocument(doc, themeId) that:
updates fills/strokes/text colors based on theme map
re-renders cover/dividers
preserves text content and page order
This way theme switching is instant and doesn’t rerun AI.

5) The one thing that causes 80% of failures
If your AI returns non-JSON or partially structured text, your compiler will degrade into “manual design again.”
So enforce:
strict JSON
server-side validation (zod or equivalent)
retry once with “fix JSON” prompt if parse fails
