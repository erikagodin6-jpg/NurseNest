You are implementing a global translation-system hardening fix across the NurseNest production codebase.

PROBLEM

Raw i18n keys are appearing in production UI instead of human-readable text, for example:

- hero.featureStrip.activeRecall
- hero.featureStrip.spacedRepetition
- hero.featureStrip.clinicalDecision
- hero.featureStrip.examBlueprint
- intlNursing.hub.title
- intlNursing.hub.badge
- hero.trustIndicator.*
- home.heroFeatures.*
- and similar unresolved translation keys

This is no longer a one-component bug. This is a system-level reliability issue in the localization layer.

GOAL

Create a robust, global i18n solution so unresolved translation keys never appear visibly in production again.

The fix must:
- preserve multilingual support
- keep proper namespace-based translations
- allow English fallback
- prevent raw keys from rendering in UI
- work across homepage, nursing tiers, allied health, new grad, international nursing, admin pages, and future pages
- make missing keys easier to detect during development
- avoid fragile one-off component patches

DO NOT:
- do not remove localization support
- do not hardcode all strings directly in components
- do not create multiple competing fallback systems
- do not silently swallow translation issues without logging them in development
- do not break SSR/hydration
- do not make production show raw translation keys under any circumstances

==================================================
PART 1 — AUDIT THE I18N ARCHITECTURE
==================================================

Audit the current localization implementation end to end:

- i18n initialization/config
- translation provider setup
- namespace loading
- page-level namespace registration
- locale JSON structure
- fallback language config
- server/client hydration behavior
- route-based lazy loading
- translation hook usage
- helper utilities wrapping t()

Identify all current failure modes, including:
- wrong namespace used by component
- namespace not loaded for a route
- missing key in locale file
- key exists under wrong nesting
- inconsistent casing/path naming
- async translation loading race conditions
- fallback language not configured correctly
- direct rendering of unresolved `t(key)` return values
- components assuming that the translation library never returns the raw key

==================================================
PART 2 — CREATE A SAFE GLOBAL TRANSLATION HELPER
==================================================

Implement a single reusable helper for all UI translation calls.

Examples of acceptable patterns:
- safeT(...)
- translateWithFallback(...)
- useSafeTranslation(...)

This helper must:
1. attempt the requested key in the requested namespace
2. if missing, attempt English fallback
3. if still missing, use a provided plain-language fallback/defaultValue
4. if no defaultValue exists, generate a clean human-readable fallback instead of returning the raw key
5. in development only, log a warning with the missing key, namespace, and component/page context
6. in production, never expose the raw key string to the user

Behavior requirement:
- production UI must never show values like `hero.featureStrip.activeRecall`
- development must still make it obvious that a translation is missing

==================================================
PART 3 — DEFINE FALLBACK RULES
==================================================

Implement fallback priority consistently:

Priority order:
1. requested locale key
2. English locale key
3. explicit defaultValue passed by component
4. humanized fallback derived from last key segment

Examples:
- `hero.featureStrip.activeRecall` -> "Active Recall"
- `intlNursing.hub.ctaPrepare` -> "Prepare"
- `home.heroFeatures.labValues` -> "Lab Values"

Humanization should:
- extract the last meaningful segment
- split camelCase
- replace dots/underscores/hyphens
- title-case the result
- avoid ugly raw developer strings in UI

Do not let the raw full key path appear visually.

==================================================
PART 4 — ADD A SAFE HOOK FOR COMPONENTS
==================================================

Create or update a hook so components can do something like:

const { safeT } = useSafeTranslation('home')

Then use:
safeT('hero.featureStrip.activeRecall', 'Active Recall Learning')

or equivalent.

Requirements:
- namespace-aware
- supports default value
- supports interpolation
- supports pluralization if current system supports it
- works in client and SSR contexts
- minimal refactor burden for components
