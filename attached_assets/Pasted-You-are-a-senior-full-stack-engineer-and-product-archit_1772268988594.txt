You are a senior full-stack engineer and product architect. Implement an automated daily content pipeline for NurseNest.

Goal:
For EACH tier (rpn, rn, np), generate BOTH:
1) Exam questions
2) Flashcards

Daily targets:
- Until the tier’s bank reaches 4000 items:
  - Generate 100 NEW exam questions per tier per day
  - Generate 100 NEW flashcards per tier per day
- After the tier’s bank reaches 4000 items:
  - Generate 25 NEW exam questions per tier per day
  - Generate 25 NEW flashcards per tier per day

Clarify “bank reaches 4000 items”:
- Implement this as per-tier counts:
  - question_bank count WHERE tier=’rpn’ AND status in (approved|published) >= 4000 => reduced mode for rpn questions
  - same for rn, np
- Do the same for flashcards:
  - flashcard_bank count WHERE tier=’rpn’ AND status in (approved|published) >= 4000 => reduced mode for rpn flashcards
- Make thresholds configurable in server config:
  - THRESHOLD_COUNT=4000, HIGH_RATE=100, LOW_RATE=25

Non-negotiable rules:
1) No AI calls on public page load or browsing. Generation runs only as scheduled jobs or explicit user/admin actions.
2) AI outputs are never auto-published directly.
3) Every generated item must pass a clinical verification gate before it can be published.
4) The job must be idempotent: reruns on the same date must not create duplicates.
5) Cost controls: daily budget cap + per-user/per-system rate limits.

Architecture requirements:

A) Data model
Create or confirm tables:

1) exam_questions
- id
- tier (rpn|rn|np)
- topic_tag
- difficulty (easy|med|hard)
- question_type (mcq|sata|priority|ordering)
- stem
- options_json
- correct_json
- rationale
- references_json (citations/guidelines)
- status (draft|needs_review|approved|rejected|published)
- content_hash (unique)
- created_at

2) flashcards
- id
- tier (rpn|rn|np)
- topic_tag
- front
- back
- tags_json
- references_json
- status (draft|needs_review|approved|rejected|published)
- content_hash (unique)
- created_at

3) generation_jobs
- id
- run_date (YYYY-MM-DD)
- content_type (exam_questions|flashcards)
- tier (rpn|rn|np)
- target_count
- generated_count
- mode (high_rate|low_rate)
- topic_plan_json (distribution for the run)
- status (queued|running|done|failed|partial)
- cost_estimate_json
- created_at, completed_at

4) verification_reports
- id
- entity_type (exam_question|flashcard)
- entity_id
- verdict (pass|pass_with_edits|fail|needs_human_review)
- confidence_score (0–1)
- issues_json (severity, category, fix)
- citations_json
- checked_at
- model_version

5) ai_cache
- cache_key (hash of normalized prompt/input)
- output_json
- created_at

Deduplication:
- content_hash should be a stable hash of (tier + normalized stem/front/back + correct + key tags).
- Enforce unique index on content_hash per table.

B) Scheduler
Implement a daily scheduler (America/Toronto):
- Runs once per day at a configured time (e.g., 02:00).
- Creates generation_jobs for each tier and each content_type.

Total jobs per day: 3 tiers * 2 content types = 6 jobs.

C) Target calculation logic (core requirement)
For each (tier, content_type):
1) Get published/approved count for that tier and type.
2) If count < THRESHOLD_COUNT:
   target = HIGH_RATE (100)
   mode = high_rate
   Else:
   target = LOW_RATE (25)
   mode = low_rate

D) Topic distribution (avoid 100 items on one topic)
Implement a topic pool per tier per content type:
- A weighted rotation across systems (cardio/resp/neuro/endo/renal/GI/heme/ID/OB/peds/mental health/pharm/emergency/etc.)
- Ensure daily run is balanced (e.g., max 15% from any single topic).
- Save the selected topic plan in generation_jobs.

E) AI generation (button-triggered by scheduler only)
Generate in batches (e.g., 20 at a time) until target_count reached:
- For exam questions:
  - Must include stem, options, correct, rationale, and references_json.
  - Must be exam-style and clinically accurate.
- For flashcards:
  - Must be concise, high-yield, clinically accurate.
  - Include references_json or “source type” pointers.

Use caching:
- Before calling AI, check ai_cache for identical request input_hash.
- Store results in ai_cache.

F) Clinical verification gate (mandatory)
After generation, run a server-side verification step:
- Flag:
  - unsafe dosing, incorrect triage priorities, wrong lab ranges, contraindication errors
  - scope-of-practice errors
  - hallucinated drugs/claims
  - missing references for high-risk claims
- Thresholds:
  - If verdict=fail OR confidence < 0.75 OR any severity=unsafe/incorrect -> status = needs_review (blocked from publish)
  - If verdict=pass -> status = draft (or approved if you explicitly want auto-approve; default is draft)

G) Admin review UI
Add admin pages:
- /admin/qbank/review
- /admin/flashcards/review
Filters:
- tier, topic, verdict, status, date
Actions:
- edit item
- approve
- reject
- bulk approve (only verdict=pass)
- export

H) Publishing rules
Only approved items can be published.
Public site shows only published (or approved) based on your current design.

I) Controls / safety / cost
- Implement system-wide daily budget cap.
- Implement retry logic with backoff.
- Mark job partial if cap reached.
- Log generation counts, verification pass rates, and cost estimates.

J) Manual run + testing
- Add an admin button to “Run today’s generation now” for one tier and one content_type with a small count (e.g., 5) for testing.
- Provide a test checklist and proof logs.

Deliverables:

  - how dedupe works

  - how to run manually for testing
