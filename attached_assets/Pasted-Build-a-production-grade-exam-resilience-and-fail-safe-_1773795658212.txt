Build a production-grade exam resilience and fail-safe system for NurseNest so paying users never completely lose access when one exam route, API call, or renderer fails.

PRIMARY GOAL
A subscriber must always have some usable path forward even if a live exam page breaks.

This is not just a bug fix. This is a platform reliability upgrade.

CORE REQUIREMENTS

1. PREVENT FULL EXAM ACCESS FAILURE
Implement layered protection so a single frontend, backend, or data issue cannot fully block subscribers from studying.

2. ADD BACKUP ACCESS MODES
If the normal live exam experience fails, automatically offer a fallback study experience instead of a crash page.

3. PROTECT PAYING CUSTOMERS
Subscribers must always receive an alternate usable version of the content they paid for.

BUILD THIS SYSTEM IN 7 LAYERS

LAYER 1 — EXAM DATA VALIDATION BEFORE PUBLISH
Add strict pre-publication validation for every exam.

Do not allow an exam to be published unless it passes required checks:
- valid exam id
- title exists
- status is valid
- questions array exists
- minimum required question count
- every question has supported type
- stem/content exists
- answer structure is valid
- options exist where needed
- rationale structure valid when required
- language fields valid
- no malformed JSON blobs
- no broken media references that crash rendering

If validation fails:
- block publish
- show exact error list in admin
- mark exam as “needs repair”
- do not expose broken exam to users

LAYER 2 — SAFE RUNTIME NORMALIZATION
Before rendering an exam, pass all fetched exam data through a strict normalization layer.

Requirements:
- sanitize null/undefined fields
- normalize legacy content shapes
- coerce older question formats to current schema where possible
- remove invalid questions safely
- log invalid records
- compute validQuestionCount after normalization

If some questions are bad:
- skip only those questions
- continue rendering the exam if enough valid questions remain

If too many questions are bad:
- automatically switch to fallback mode instead of crashing

LAYER 3 — QUESTION-LEVEL FAULT TOLERANCE
One bad question must never crash the whole exam.

Wrap each question renderer with defensive guards:
- unsupported type -> render fallback question card
- missing options -> render fallback question card
- malformed rationale -> hide rationale area but keep question usable if possible
- media failure -> show question without media
- translation failure -> fall back to default language if available

Fallback question card text:
“This question is temporarily unavailable. Please continue to the next question.”

Exam session must remain usable.

LAYER 4 — EXAM-LEVEL FALLBACK MODES
If the live interactive exam player fails, automatically provide alternate study access.

Build these fallback modes in priority order:

A. Safe Exam Player
- minimal stripped-down exam renderer
- plain text question + options
- no advanced animations
- no heavy client dependencies
- no advanced widgets unless validated
- works even if enhanced exam UI fails

B. Study Mode Fallback
- show all valid questions in a simple scrollable study format
- display questions one by one or in a simple list
- allow answer reveal and rationale reveal
- preserve user access to the content

C. Printable/Static Backup
- generate or serve a static HTML/PDF-safe backup version of each published exam
- include title, questions, answer key, rationales where allowed
- accessible only to entitled users
- available if live API or live player fails

D. Backup Practice Set
- if a specific exam is broken and cannot be safely rendered, automatically offer a substitute:
  “This exam is temporarily unavailable. Your access is protected. Start a backup practice set now.”
- provide a comparable backup exam from same tier, exam type, profession, and language where possible

LAYER 5 — SUBSCRIBER PROTECTION UX
Replace generic crash screens with a subscriber-safe recovery screen.

If exam cannot fully load, show:
- “We’re having trouble loading this exam.”
- “Your access is protected.”
- buttons:
  - Retry exam
  - Open Safe Exam Player
  - Open Study Mode
  - Open Printable Backup
  - Launch Backup Practice Set
  - Return to Dashboard

Also include:
- incident id
- support/report issue button
- automatic error logging

Do not ever leave users on a blank page or generic React error boundary for recoverable exam failures.

LAYER 6 — BACKEND AND CONTENT REDUNDANCY
Create durable backup access architecture.

Requirements:
- store normalized published exam snapshots
- create immutable published-version records
- preserve last known good version of each exam
- if newest version fails validation or rendering, automatically serve last known good version
- keep a backup rendered payload separate from raw generator payload
- cache exam payloads for fast failover
- support serving published exams from cached snapshot if live transform fails

Build “last known good exam version” logic:
- every successful published exam version gets archived
- runtime can fall back to latest verified good version
- admin can restore a known-good version with one click

LAYER 7 — MONITORING, ALERTING, AND ADMIN RECOVERY
Add observability and fast-recovery tools.

Capture and log:
- exam id
- user id if available
- tier
- profession
- language
- route
- fallback mode triggered
- question id causing issue
- stack trace
- API response status
- timestamp

Build admin tools:
- broken exam detection dashboard
- list of exams triggering fallback mode
- one-click disable broken exam
- one-click swap to backup version
- one-click restore last known good version
- content health warnings before users report problems

Add proactive alerting:
- if exam load failures cross threshold, notify admin
- if a published exam returns zero valid questions, notify admin immediately
- if fallback mode usage spikes, flag incident

ADDITIONAL RELIABILITY REQUIREMENTS

1. Graceful access degradation
Users should lose only the failing feature, not the entire exam experience.

2. Entitlement preservation
If a subscriber has valid access, every fallback mode must respect their entitlement and remain available.

3. Minimal dependency backup path
Create a lightweight fallback player with as few dependencies as possible so it continues working even if the main exam UI fails.

4. Circuit breaker behavior
If a specific exam repeatedly fails, stop routing users into the crashing experience and automatically route them to:
- last known good version, or
- safe exam player, or
- backup practice set

5. Session recovery
If a user was mid-exam and the app crashes:
- preserve session state where possible
- restore progress on retry
- otherwise offer restart in safe mode

6. Static backup generation
For every published exam, create a static backup artifact or cached backup payload at publish time so backup exists before failures occur.

7. Regression tests
Add tests for:
- malformed exam payload
- missing questions
- unsupported question type
- failed media asset
- i18n mismatch
- live player crash
- fallback mode routing
- serving last known good version
- subscriber retained access during failure
