- global variables holding datasets
- caches without size limits
- stale sessions not being cleaned up

SECTION 2 — MEMORY GUARDRAILS
==================================================

Implement a central memory protection module.

Requirements:
- sample memory usage on an interval
- define thresholds:
  - warning mode at 70%
  - protection mode at 80%
  - critical mode at 88%
- expose current system health internally
- write structured logs when thresholds are crossed
- prevent log spam with rate limiting

When warning mode starts:
- log warning
- mark system status as degraded

When protection mode starts:
- automatically reduce payload sizes
- reduce concurrency on expensive routes
- disable nonessential background jobs
- pause bulk AI generation
- pause precomputation jobs
- force pagination everywhere possible

When critical mode starts:
- activate SAFE MODE
- keep login, billing verification, study access, and active exams alive
- temporarily disable nonessential heavy features
- reject oversized generation jobs with friendly fallback messaging
- preserve paying-user access to core study functionality

==================================================
SECTION 3 — EXAM ENGINE HARDENING
==================================================

Refactor the exam system so it is memory-safe.

Rules:
- NEVER load an entire exam into memory if not required
- NEVER send full exam datasets to the client at once
- NEVER store large exam payloads in session memory

Instead:
- store only exam metadata, question IDs, and user answers
- fetch questions on demand in small batches
- paginate or stream exam items
- preload at most a very small next-question window
- release previous question objects after use where possible

Add:
- safe resume support
- partial-progress persistence
- fallback recovery if an exam fetch fails
- retry with capped backoff
- user-friendly message if a question batch cannot load
- automatic reload of just the missing batch, not the whole exam

==================================================
SECTION 4 — API RESPONSE CONTROLS
==================================================

Add response-size protections to all heavy endpoints.

Requirements:
- set sane max page sizes
- cap number of questions returned per request
- cap number of lessons/cards/blog items returned per request
- reject unbounded queries
- enforce field selection where possible
- avoid returning giant nested objects
- compress responses if supported
- add defaults even if client forgets pagination

Apply this to:
- exam endpoints
- question bank endpoints
- lesson endpoints
- flashcard endpoints
- admin content endpoints
- AI generation endpoints
- analytics endpoints

==================================================
SECTION 5 — CACHE + SESSION SAFETY
==================================================

Audit all caches and session storage.

Implement:
- max-size cache limits
- TTL expiration
- cleanup of stale entries
- no unlimited in-memory maps
- no indefinite retention of generated content blobs
- no storage of full user-content objects in memory when IDs will do

For sessions:
- remove stale exam sessions
- clean abandoned sessions
- minimize session footprint
- persist only necessary data
- move large state out of memory if currently stored there

==================================================
SECTION 6 — FRONTEND PROTECTION
==================================================

Refactor the frontend to avoid memory-heavy rendering.

Requirements:
- no rendering of massive lists without virtualization or pagination
- no storing huge datasets in React state
- no duplicate fetch loops
- no uncontrolled useEffect calls
- no repeated polling without cleanup
- lazy load heavy views
- only fetch what is needed for the visible screen
- use skeleton states and incremental loading
- guard against repeated retries when backend is degraded

Add:
- safe degraded-mode banners
- user-friendly messaging like:
  “We’re experiencing heavy system load. Core studying remains available.”
- preserve trust and clarity without exposing internals
