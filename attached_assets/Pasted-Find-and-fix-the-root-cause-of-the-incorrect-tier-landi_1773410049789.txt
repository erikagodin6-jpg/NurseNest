Find and fix the root cause of the incorrect tier landing page messaging across the live NurseNest production site.

Problem:
Multiple tier landing pages are incorrectly inheriting or displaying REx-PN-focused marketing copy even when the page is for RN, NP, or other non-RPN audiences. This is a shared configuration/component/content inheritance issue and needs to be fixed properly at the system level, not with one-off text edits.

Non-negotiable:
Work only against the live published production database, production CMS/content source, and production config used by the public NurseNest site. Do not write to dev-only, preview-only, mock, or temporary Replit data. Before editing anything, verify the app is pointed to the same production environment used by the real public site.

Main objective:
Fix the tier-aware landing page architecture so every tier page automatically renders the correct marketing copy, exam references, calls to action, and product language based on its actual tier.

STEP 1 — FULL ROOT-CAUSE AUDIT

Audit the entire landing page rendering path and identify exactly why non-RPN pages are showing REx-PN messaging.

Inspect all possible sources of the issue, including:
- shared hero components
- shared CTA components
- tier config files
- content mapping objects
- default page props
- page templates
- CMS/database content fallbacks
- i18n translation keys
- server-side loaders
- client-side hydration overrides
- environment-based config differences
- static JSON seed data
- SEO metadata templates
- schema/structured data generators
- card/reusable promo sections
- FAQ/review/testimonial component defaults

Find whether the issue is caused by one or more of the following:
- a global hardcoded REx-PN default
- missing tier-specific copy causing fallback to REx-PN
- incorrect slug-to-tier mapping
- bad enum/type mapping
- broken content lookup
- stale cached content
- preview content being mixed with production
- shared translation keys reused incorrectly
- a UI component that assumes all exam pages are PN by default

Do not stop at symptom-level fixes. Identify the real architectural cause.

STEP 2 — CREATE A PROPER TIER-AWARE CONTENT SYSTEM

Implement a centralized, clean tier-aware content configuration system so marketing copy is generated by tier instead of duplicated or hardcoded across components.

Create or refactor to a structure like this conceptually:
- each tier has its own content object/config
- each page pulls from the appropriate tier content based on route/slug/page type
- no page should fall back to REx-PN unless the actual tier is RPN/PN
- shared components should accept tier-aware props instead of using generic default strings

The system should support at minimum:
- RPN / LVN
- RN / NCLEX
- NP Advanced
- any existing allied health, specialty, transition-to-practice, or role-specific pages already in the site

Tier content model should include:
- hero headline
- hero subheadline
- CTA text
- support text
- exam/product references
- learner identity language
- feature section copy
- FAQ intros
- trust/review snippets if applicable
- SEO title
- meta description
- alt text where relevant
- badge labels
- schema fields if generated dynamically

STEP 3 — DEFINE CORRECT MESSAGING BY TIER

Update the content logic so messaging is accurate by audience.

RPN / LVN pages should reference:
- REx-PN
- practical nursing
- PN/LVN exam prep if supported
- practical nurse confidence and readiness

RN pages should reference:
- RN-level prep
- NCLEX-RN where applicable
- Canadian RN preparation where relevant
- advanced med-surg, critical thinking, prioritization, and RN clinical judgment

NP pages should reference:
- advanced practice
- board prep / certification readiness
- advanced assessment, diagnostics, prescribing/pharmacology, and clinical reasoning
- the NP exam families/tracks you support where applicable

Other specialty, floor, or role pages should:
- market the actual audience for that page
- not inherit REx-PN copy unless that page is explicitly PN-focused

STEP 4 — FIX ALL SHARED COMPONENTS

Refactor shared components so they do not inject REx-PN copy by default.

Specifically audit and fix:
- hero section component
- CTA banners
- feature cards
- comparison sections
- testimonials/reviews modules
- FAQs
- page intro sections
- tier badges
- exam labels
- category ribbons
- SEO head/meta builder
- schema markup generator
- reusable lesson/product marketing sections
- empty states and fallback text
- search/no-result text if tier-specific

Requirements:
- shared components must receive explicit tier-aware props
- remove unsafe generic defaults that say REx-PN
- if data is missing, fail gracefully with neutral copy or a clearly tier-safe fallback, not PN-specific language
- make the fallback hierarchy explicit and safe

STEP 5 — FIX ROUTING, SLUGS, AND MAPPINGS

Audit page routing and tier resolution logic.

Make sure:
- each landing page route correctly maps to its intended tier
- slugs resolve to the correct tier config
- dynamic routes do not default to RPN when tier data is missing
- if a page type is reused, its resolved marketing copy still comes from the correct tier object
- route params, server loaders, and front-end rendering agree on page identity

If there is any slug-to-tier mismatch, fix it safely without breaking URLs or SEO.

STEP 6 — FIX SEO AND STRUCTURED DATA

Audit and correct SEO output for all tier landing pages.

Fix:
- meta titles
- meta descriptions
- canonical tags
- Open Graph descriptions if generated
- alt text containing wrong exam language
- schema/structured data referencing the wrong exam audience

Make sure RN pages do not have REx-PN metadata.
Make sure NP pages do not have REx-PN metadata.
Make sure specialty pages do not inherit incorrect PN metadata.

STEP 7 — FIX I18N / TRANSLATION FALLBACKS

If any multilingual translation layer is involved, audit whether translation keys are falling back to PN strings.

Check:
- missing translation keys
- shared namespace collisions
- fallback locale behavior
- reused exam strings in translation JSON
- hydration differences between translated and non-translated pages

Refactor so translations are tier-aware and safe. Missing translations should not force the page to display REx-PN language.

STEP 8 — ADD VALIDATION AND ADMIN SAFEGUARDS

Add safeguards so this never happens again.

Create validation that checks:
- each tier landing page has required tier-specific marketing fields
- no RN/NP/non-PN page contains unintended REx-PN copy in critical sections
- no page is published if it fails tier messaging validation
- SEO metadata matches the resolved tier
- shared config changes do not break all tiers silently

Add an admin audit view showing:
- page name
- resolved tier
- hero headline
- CTA text
- metadata preview
- validation status
- flags for incorrect exam references
- one-click open/edit

If possible, add automated tests for:
- route resolves correct tier
- hero copy matches correct tier
- metadata matches correct tier
- fallback logic never defaults to REx-PN for RN/NP pages

STEP 9 — VERIFY EVERY LANDING PAGE ON THE LIVE SITE

After fixes, manually and programmatically verify the actual live public pages, not just Replit preview.

Check at minimum:
- RPN / LVN landing page
- RN landing page
- NP landing page
- all other active tier/specialty landing pages already in production

Confirm:
- headline is correct
- subheadline is correct
- CTA is correct
- feature cards are correct
- FAQ intro is correct
- metadata is correct
- schema is correct
- no stray REx-PN references remain on non-RPN pages

Refresh/rebuild caches if needed so production reflects the fixed content.

STEP 10 — REPORTING

When complete, generate a concise dev/admin report that includes:
- exact root cause(s) found
- which components/configs were responsible
- which routes/pages were affected
- which files/modules/data structures were changed
- whether the issue involved hardcoded defaults, missing config, wrong slug mapping, i18n fallback, cache, or multiple causes
- confirmation that all tier landing pages now resolve correct messaging on the live site
- confirmation that unsafe REx-PN fallback behavior was removed

IMPORTANT GUARDRAILS
- Do not apply one-off text replacements without fixing the shared system
- Do not leave hidden PN defaults in shared components
- Do not fix only visible hero text while leaving wrong metadata/schema in place
- Do not make changes only in preview/dev
- Do not break URLs, SEO structure, or paywall logic
- Do not overwrite intentional tier-specific messaging with generic bland copy

SUCCESS DEFINITION
This task is complete only when:
- the root cause of incorrect REx-PN inheritance is fixed at the architectural/shared-component level
- each tier landing page renders its own correct exam and learner messaging
- shared components are safely tier-aware
- metadata and schema also match the correct tier
- the live public site reflects the corrected behavior
