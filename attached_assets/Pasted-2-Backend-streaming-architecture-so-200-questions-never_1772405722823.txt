2) Backend streaming architecture (so 200 questions never crashes)
Goal

Stop trying to get 200 items in one response. Instead:

Generate in chunks (e.g., 10–25 items)

Validate each chunk

Persist immediately

Resume safely if the model times out or produces malformed JSON

Core design

You are building a QBank ingest pipeline, not a document generator.

Chunking strategy

Chunk size: 10–15 for SATA-heavy sets (safer)

Each chunk request includes:

“You must output EXACTLY X items”

“Start ids from next_idx”

A compact “state” block that tells the model what’s already done (counts + recent stems to avoid repeats)

Streaming / incremental generation API

POST /api/generations
Creates a generation job:

{
  "template": "question_pack",
  "targetCount": 200,
  "chunkSize": 15,
  "promptBase": "...rules..."
}

POST /api/generations/:id/run
Starts or resumes generation.

GET /api/generations/:id/status
Returns:

status

created_count

validation errors

latest event

Worker loop (the important part)

Pseudo-logic:

Load generation

While created_count < target_count

Build chunk prompt with:

next_idx

chunk_size = min(chunkSize, target_count - created_count)

distribution “needed” (e.g., SATA shortfall, systems shortfall)

anti-duplication list (hashes or last 20 stems)

Call model

Parse JSON only

Validate each item

Save valid items

If invalid:

reject only invalid items

re-request “REPLACE these ids only” (surgical retry)

Update state + counts

This prevents “Pipeline step content failed” because the generator is never holding a giant payload.

The #1 trick that makes parsing reliable

Require the model to return:

{ "items": [ ... ], "meta": { "chunk": 3, "startId": 46, "count": 15 } }

Not a huge full bundle.

You assemble the bundle in your DB.

Duplicate detection

Compute a normalized hash:

lowercase stem

remove punctuation + extra spaces

optionally remove numbers

Store it in questions.hash unique.
If insert fails (duplicate), request replacement for that id.

Resume safety

If the run stops at 143/200, the next run simply reads:

created_count=143

prompt_state distributions
Then continues from 144.

4) Fix the generator validation logic (so it stops failing + stops producing 5 questions)

This is where your platform is likely breaking right now.

Validation must be “partial accept” not “all-or-nothing”

Current behavior sounds like:

if output isn’t perfect → fail whole step → you get almost nothing (like 5 questions)

Instead:

Accept valid items, reject invalid ones, and request replacements.

Validation rules (strict but practical)

For each item:

Required fields

stem non-empty, min length e.g. 40 chars

type in {mcq, sata}

difficulty in {moderate, hard, very_challenging}

system in approved set

choices array exists

Choice rules

MCQ: exactly 4 choices labeled A-D

SATA: min 5 choices labeled A-E (allow up to H)

No duplicate choice texts

No empty choices

Correct answers rules

MCQ: exactly 1 answer

SATA: 2–5 answers (set your max; prevents nonsense)

Each correct answer label must exist in choices

Rationale rules

correctReasoning required

incorrectBreakdown must contain all labels for that item

keyPathophysiology required

nursingImplication required

examPearl required (or allow fallback generation)

Anti-duplication

hash stem → must be unique within generation

Replacement protocol (this is the key)

If you requested ids 46–60 and 3 items are invalid:

Send model a corrective prompt:

“Return JSON with replacements for ids: 49, 53, 58 only”

“Do not rewrite other ids”

“Keep distribution the same”

“Ensure SATA has ≥5 options”

This keeps progress stable and prevents the “start over” spiral.

Token-budget guardrails (prevents runaway)

Before calling the model:

compute remaining budget

cap chunk size based on:

average tokens per question type

SATA is heavier than MCQ

If chunk response exceeds limits:

reduce chunk size automatically and retry.

Why you’re seeing “only 5 questions”

Common causes:

Your validator rejects most items and your code discards everything

Your JSON parser fails midway and only reads the first valid subset

Your schema expects a different key (e.g., items vs questions)

Your prompt conflicts with template expectations (Cram Guide expects blocks; you return JSON)

The partial-accept + replacements pattern fixes #1 and #2. Hybrid architecture fixes #4.
