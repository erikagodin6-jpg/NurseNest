CRITICAL DATABASE AND PUBLISHING SAFETY RULE FOR NURSENEST

The system must never confuse preview/development data with the live published site.

Implement strict environment-aware database and publishing protections so that content generated or edited from the admin dashboard can be intentionally sent to the correct target environment and never silently written only to development when production was selected.

====================================================
ENVIRONMENT VISIBILITY
====================================================

Add a permanent, highly visible environment badge in the admin header and all AI Ops pages:

- DEVELOPMENT
- STAGING
- PRODUCTION

This badge must be impossible to miss.

Also display:
- current deployment target
- current database target
- safe database fingerprint/hash identifier
- current app mode

Do not expose secrets, but provide enough information to clearly verify which environment/database is active.

====================================================
EXPLICIT TARGET SELECTION
====================================================

For all generation, publishing, bulk edits, and admin AI actions, require an explicit target selector:

- DEVELOPMENT
- STAGING
- PRODUCTION

Rules:
- default to DEVELOPMENT or DRAFT mode for safety
- PRODUCTION must never be assumed implicitly
- if PRODUCTION is selected, require explicit confirmation
- store the selected target with the job/action record

====================================================
PRODUCTION CONFIRMATION RULE
====================================================

When the admin selects PRODUCTION for any of the following:
- content generation
- publish actions
- bulk metadata writes
- approved AI content saves
- approved admin content edits
- approved patch-related content changes

Show a confirmation modal that includes:
- selected target = PRODUCTION
- environment badge
- database fingerprint
- summary of what will be written
- count of affected items
- approval checkbox
- confirmation button

No production write should proceed without this confirmation.

====================================================
DATABASE TARGET VERIFICATION
====================================================

Before any write job begins, run a preflight verification step that checks:
- active environment
- selected target
- database connection target
- deployment mode
- write permissions
- safe database fingerprint

If the selected target and actual database target do not match:
- block the write
- log the mismatch
- show a clear admin error
- do not silently fall back to development

Example:
If the admin selected PRODUCTION but the app is connected to development DB, block the job and show:
“Production target selected, but current database connection is not production. Write blocked.”

====================================================
NO SILENT FALLBACKS
====================================================

Under no circumstances should the system:
- silently write to development when production was selected
- silently publish only to preview when production was selected
- silently switch targets
- silently downgrade a production write to a draft save without clearly telling the admin

All target downgrades must be explicit and visible.

====================================================
AUDIT LOGGING
====================================================

Log every generation, edit, publish, and bulk action with:
- admin user
- selected target
- actual environment
- actual database fingerprint
- content type
- item count
- provider/model used if AI involved
- action type
- approval state
- timestamps
- success/failure state
- mismatch/block reason if any

Create a dedicated audit log page in admin:
Publishing & Environment Logs

====================================================
PUBLISHED SITE RULE
====================================================

Implement a clear distinction between:
- saved draft in admin
- validated but unpublished content
- approved content
- published live content

Add visible status badges:
- DRAFT
- VALIDATED
- APPROVED
- PUBLISHED LIVE

Do not label something as published unless it has actually been written to the production target successfully.

====================================================
POST-WRITE VERIFICATION
====================================================

After every production publish/write, perform a post-write verification step:
- confirm records exist in the production target
- confirm publish status updated correctly
- confirm item counts match expected results
- log verification result

If post-write verification fails:
- mark action as FAILED_VERIFICATION
- alert admin
- do not present the action as successfully published

====================================================
PREVIEW VS LIVE CLARITY
====================================================

In admin UI, clearly differentiate:
- Preview-only changes
- Draft-only changes
- Live published changes

Every content item should show:
- current environment source
- current publish status
- last published timestamp
- last published by
- target destination

====================================================
SAFE WRITE ARCHITECTURE
====================================================

All content writes must go through one centralized write service, for example:

EnvironmentAwareContentWriteService

This service must:
- accept selected target
- verify current environment
- verify DB fingerprint
- block mismatches
- execute write
- verify result
- log audit trail

No direct scattered writes should bypass this service.

====================================================
ADMIN BULK ACTION SAFETY
====================================================

For bulk actions:
- show exact number of records affected
- show selected target
- require confirmation for production
- allow dry-run preview before applying
- log all results

Add dry-run mode for:
- bulk metadata updates
- bulk publishing
- bulk AI-approved content saves
- bulk retagging
- bulk SEO updates

====================================================
DEPLOYMENT/PUBLISHING CHECK
====================================================

Add an admin diagnostic tool called:
Environment & Publish Check

This tool should test:
- current app environment
- current database target
- production write readiness
- audit logging readiness
- publish verification pipeline
- last successful production publish

====================================================
NON-NEGOTIABLE RULES
====================================================

- Never silently write to the wrong database
- Never let production selection write only to preview/dev
- Never show content as live if it is not actually live
- Never bypass preflight verification
- Never bypass post-write verification
- Always make environment and target visible
- Always log every important write/publish action
