You are working on the NurseNest production codebase.

We already confirmed:
- Build time is fast (~66 seconds total)
- Publish time is extremely slow (30–50 minutes)
- The bottleneck is NOT the frontend build
- The likely bottlenecks are:
  1. npm install time during deploy
  2. oversized server bundle (~28–29MB)
  3. startup code blocking readiness
  4. seed data / question banks / generated content being bundled or loaded at boot

Your mission:
Perform a deep architectural fix for bundle size, startup time, and deployment readiness so publish time drops to under 5 minutes and the app becomes healthy almost immediately.

==================================================
PRIMARY GOALS
==================================================

1. Reduce server bundle size dramatically
2. Remove all startup-blocking logic
3. Prevent deployment from waiting on unnecessary work
4. Keep all existing core user functionality working
5. Do not introduce regressions to auth, payments, entitlements, exams, flashcards, lessons, or admin tools

==================================================
PHASE 1 — FULL DEPLOYMENT PATH AUDIT
==================================================

Identify and document the exact deploy path:

- dependency install command
- build command
- server run command
- entrypoint
- startup initialization chain
- health check route
- anything executed before app.listen()
- anything executed immediately after app.listen()
- background jobs started inside the web process
- cache warmers
- content preloaders
- seed/repair scripts
- migration/backfill scripts

Output a concise summary of:
- what runs during install
- what runs during build
- what runs during startup
- what blocks readiness

==================================================
PHASE 2 — FIND SERVER BUNDLE BLOAT
==================================================

Audit what is contributing to the ~28–29MB server bundle.

Specifically search for:
- imported JSON data files
- question banks embedded in source
- seed content bundled into server code
- generated lesson/question/flashcard content in repo
- large constants
- static assets imported into server
- admin datasets or templates bundled into runtime
- backup files or dead files imported indirectly

Find every large import chain feeding the server bundle.

For each large contributor, classify it as one of:
- must stay in runtime
- should be lazy-loaded on demand
- should move to database
- should move to external storage
- should be removed from server bundle entirely

Then implement the fixes.

==================================================
PHASE 3 — REMOVE ALL STARTUP BLOCKERS
==================================================

The app must bind to PORT immediately.

Before app.listen(), there must be NO:
- loading all questions into memory
- loading all exams into memory
- content audits
- sitemap generation
- AI generation jobs
- full DB scans
- backfills
- queue warmup
- cache warmup
- repair scripts
- file system crawls
- preloaders for lessons/flashcards/mock exams/categories
- analytics sync jobs
- billing reconciliation
- translation validation sweeps

If any of this exists, move it out.

Acceptable alternatives:
- lazy load on first use
- scheduled worker
- admin-triggered maintenance job
- background task started after readiness and guarded so it never blocks health

==================================================
PHASE 4 — RESTRUCTURE CONTENT LOADING
==================================================

Question banks, lessons, exams, flashcards, and generated educational content must NOT be bundled into the server build if they are currently embedded as source imports.

Implement this rule:

- server contains logic
- database/storage contains content
- routes fetch only what they need
- no whole-library preload
- every content fetch must be scoped and paginated

Mandatory protections:
- no unbounded queries
- no loading full exam libraries into memory
- no loading all questions for all tiers at startup
- use LIMIT / pagination / cursor-based fetches where appropriate
- cache only small hot subsets, never the entire corpus

If any endpoint currently loads entire banks or all related items into memory, refactor it.

==================================================
PHASE 5 — INSTALL / DEPENDENCY OPTIMIZATION
==================================================

Audit package.json and lockfile usage.

Implement:
- remove unused dependencies
- move all non-runtime packages to devDependencies
- ensure production deploy installs only production dependencies if platform supports it
- remove duplicate libraries
- remove oversized libraries that are not actually needed in production runtime
- verify no build-only libraries are required by server startup

Check especially for heavy libraries related to:
- PDF generation
- image processing
- AI SDKs
- testing
- linting
- migration tooling
- CLIs
- analytics debug tooling

Output:
- removed packages
- packages moved to devDependencies
- packages still required at runtime and why

==================================================
PHASE 6 — HEALTH CHECK AND READINESS
==================================================

Create or refactor a minimal /healthz endpoint.

Requirements:
- returns immediately
- no deep DB query
- no third-party API checks
- no cache dependency
- no expensive logic
- suitable for deployment readiness

The app must become healthy as soon as the server is actually capable of serving requests.

Do not tie health to optional systems.

==================================================
PHASE 7 — SEPARATE WEB SERVER FROM MAINTENANCE WORK
==================================================

If the main web process is doing any of the following, remove them from the main process:
- cron/scheduler startup
- content generation jobs
- queue processing
- repair jobs
- cache rebuilds
- analytics aggregation
- email digests
- subscription reconciliation
- orphan cleanup
- translation scanning
- content expansion jobs

Create a clean separation:
- web process = request handling only
- worker process or manual maintenance job = heavy background tasks

If true multi-process separation is not possible in current environment, then disable those jobs during deploy/startup and gate them behind explicit invocation.

==================================================
PHASE 8 — ROUTE AND EXAM SYSTEM HARDENING
==================================================

Because exam routes have already been associated with overload/memory issues, audit all exam-related endpoints and loaders.

Find and fix:
- routes that fetch unbounded content items
- exam assembly logic that materializes too much data at once
- repeated cloning/mapping/filtering of large arrays
- per-request loading of massive full datasets
- duplicate caching layers holding the same content
- in-memory accumulation across requests
- retries that duplicate allocations
- global caches with no max size / TTL / eviction
- question expansion logic that runs on user request path
- joins or content enrichment done in memory instead of via targeted queries

Add safeguards:
- hard limits
- pagination
- defensive timeouts
- cache caps
- size-based rejection of huge operations
- graceful fallback if optional content fails

==================================================
PHASE 9 — BUNDLE-SAFE IMPORT RULES
==================================================

Enforce these rules in the codebase:

1. Server files must never import giant data blobs directly
2. JSON imports over a reasonable threshold should be rejected or flagged
3. Seed data files must never be imported by request-serving runtime code
4. Admin-only tooling must not leak into user-serving bundle
5. Any large optional module must be dynamically imported only when needed

Implement practical code changes, not just comments.

==================================================
PHASE 10 — MEASUREMENT
==================================================

Measure and report:

Before vs after:
- server bundle size
- client bundle size
- npm install time
- build time
- startup time
- time to first healthy response
- estimated publish time
- memory usage at startup

Also identify:
- top remaining bottlenecks
- any follow-up items not yet solved

STRICT REQUIREMENTS
==================================================

- Do not leave TODO comments instead of fixes
- Do not remove core features
- Do not break auth, billing, entitlements, exams, or admin
- Do not keep heavy tasks in startup
- Do not keep seed data in the runtime bundle
- Do not use unbounded queries
- Do not preload full datasets in the web process
