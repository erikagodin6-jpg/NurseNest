Refactor the entire NurseNest access-control system into a centralized entitlement-based paywall architecture so Replit AI stops accidentally removing or bypassing premium access restrictions.

PROBLEM
The current paywall is too fragile. Free users are sometimes getting full access to premium lessons, and Replit AI changes are repeatedly breaking the paywall. I need a robust, centralized system that cannot be easily bypassed or accidentally removed when other features are edited.

GOAL
Create a durable entitlement architecture that controls access to all premium content and tools across the platform, with backend enforcement as the source of truth.

PLATFORM AREAS TO PROTECT
- Lessons
- Lesson lists
- Individual lesson pages
- Flashcards
- Flashcard decks
- Question banks
- CAT exams
- Mock exams
- Analytics dashboards
- Performance reports
- Study tools
- Premium downloads
- Any future premium features

TIERS / PRODUCT AREAS TO SUPPORT
- Free
- RPN
- RN
- NP
- Allied Health
- Imaging
- New Grad
- Admin

CORE ARCHITECTURE REQUIREMENTS

1. CREATE A CENTRAL ENTITLEMENT LAYER
Build a single source of truth for feature access, for example:
- server/entitlements.ts
- shared/entitlements.ts
- lib/access-control.ts

This module should determine exactly what a user can access based on:
- role
- subscription status
- subscription tier
- purchase history if relevant
- admin status
- preview/free rules

It should return a normalized entitlement object such as:

{
  canAccessPremiumLessons: boolean,
  canAccessFlashcards: boolean,
  canAccessQuestionBank: boolean,
  canAccessAnalytics: boolean,
  canAccessMockExams: boolean,
  canAccessTier: {
    rpn: boolean,
    rn: boolean,
    np: boolean,
    allied: boolean,
    imaging: boolean,
    newgrad: boolean
  },
  previewOnly: boolean,
  isAdmin: boolean
}

2. BACKEND IS THE SOURCE OF TRUTH
Do not rely on frontend checks to protect premium content.
Every backend route that returns protected content must use the centralized entitlement layer before returning data.

Protect:
- lesson list endpoints
- lesson detail endpoints
- search endpoints
- flashcard endpoints
- question endpoints
- quiz endpoints
- analytics endpoints
- mock exam endpoints
- premium dashboard endpoints

If the user is not entitled, the backend must:
- return 403 for premium content
- or return preview-only content if explicitly allowed

3. CREATE REUSABLE GUARDS / HELPERS
Add reusable helper functions such as:
- getUserEntitlements(user)
- requireEntitlement(req, entitlementKey)
- canAccessTier(user, tier)
- isPremiumContentAllowed(user, resource)
- getPreviewContentLimit(user, contentType)

This should eliminate scattered inline logic throughout the app.

4. IMPLEMENT PREVIEW RULES EXPLICITLY
Free users should only see a limited preview, not full access.

Example preview rules:
- max 3 to 5 preview lessons per tier
- limited flashcards
- limited question access
- no analytics
- no mock exams
- no full adaptive exams

Preview rules must be defined in one central config, not hardcoded in many places.

5. ADMIN BYPASS MUST BE EXPLICIT AND SAFE
Admin users should retain full access for platform management and previewing content, but admin bypass must not leak into free-user logic.

Search for and remove unsafe logic patterns like:
- role === 'admin' || role === 'free'
- !subscription => allow access
- missing subscription check defaults to allow
- frontend-only lock components with unrestricted backend data

6. PROTECT DIRECT URL ACCESS
A free user must not be able to bypass the paywall by directly visiting:
- /lesson/:slug
- /flashcards/:deck
- /questions/:id
- /analytics
- /mock-results
- any premium route

Individual content routes must enforce entitlements server-side.

7. TIER MATCHING RULES
Users should only access content included in their tier.
Examples:
- RN subscriber can access RN content
- RPN subscriber can access RPN content
- NP subscriber can access NP content
- Allied subscriber can access Allied content
- free user can only access preview items
- admin can access all

If cross-tier access is intentionally allowed anywhere, define it explicitly in the entitlement system, not ad hoc.

8. ADD SERVER LOGGING
For all protected routes, add clear logs:
- user id
- email if available
- role
- subscription status
- subscription tier
- requested content type
- requested content tier
- entitlement result
- allow/deny outcome

Logs should help diagnose future paywall regressions quickly.

9. CREATE A SINGLE PAYWALL POLICY CONFIG
Add a central config object for policy, for example:
- preview lesson counts
- feature availability by tier
- premium feature matrix
- admin bypass rules

This config should make it easy to update access without rewriting route logic.

10. FRONTEND SHOULD MIRROR, NOT DECIDE
Update frontend components so they:
- read entitlement state from the backend or authenticated user payload
- show locked overlays, upgrade prompts, and preview labels
- never assume access without backend confirmation

Frontend can display:
- Locked lesson overlay
- Upgrade CTA
- Preview badge
- Most popular plan prompts

But backend still controls actual data access.

11. AUDIT THE WHOLE CODEBASE FOR PAYWALL BREAKS
Search the full codebase for:
- lesson routes
- flashcard routes
- question routes
- mock exam routes
- analytics routes
- admin preview logic
- role checks
- subscription checks
- isPremium
- premium
- entitlement
- tier
- access

Replace scattered fragile logic with centralized entitlement helpers wherever possible.

12. PREVENT FUTURE REPLIT AI REGRESSIONS
Add comments in the core entitlement files stating:
- This file is the source of truth for all premium access
- Do not bypass with route-local ad hoc checks
- Do not return premium data without entitlement verification
- All new premium features must integrate with this entitlement system

Where appropriate, add small tests or assertions to make regressions easier to catch.

13. ADD TEST / VALIDATION COVERAGE
Add simple validation or tests for scenarios such as:
- free user requests premium RN lesson -> denied
- free user requests preview lesson -> allowed
- RN user requests RN lesson -> allowed
- RN user requests NP lesson -> denied unless explicitly allowed
- admin requests any lesson -> allowed
- free user requests analytics -> denied
- free user directly opens premium lesson URL -> denied

14. DO NOT BREAK EXISTING STRUCTURE
Preserve:
- current routing
- current visual design
- slugs
- lesson IDs
- analytics tracking
- admin tools
- subscription billing

This is a refactor for access control, not a redesign.

15. APPLY TO ALL CONTENT TYPES, NOT JUST LESSONS
Do not stop at lessons.
Any premium educational asset or dashboard must use the same entitlement system.

16. FINAL OUTPUT REQUIRED
After implementation, return:
- which files were created or modified
- where the centralized entitlement system lives
- which routes now use it
- what preview rules are active
- what admin rules are active
- whether direct URL bypass is blocked
- any remaining routes still needing migration

DESIGN PRINCIPLES
- backend-first enforcement
- centralized access control
- explicit preview policy
- explicit admin bypass
- tier-aware permissions
- no silent default-allow behavior
- scalable for future NurseNest growth

IMPORTANT
This is a production-critical paywall stabilization task.
The final system must be much harder for future Replit AI edits to accidentally break.
Do not use scattered inline fixes. Build a centralized entitlement architecture and migrate the existing protected routes to use it.
