You are a senior full-stack engineer. Add “time-on-question” analytics to my mock exam system and display it in the user report card.

Requirements:

1) Timing capture (front-end)
- For mock exams, track time spent per question with millisecond precision.
- Start timing when a question becomes visible/active.
- Stop timing when the learner navigates away from that question (next/back/jump/submit).
- If the browser tab becomes hidden (visibilitychange) or the window loses focus (blur), pause timing; resume on focus/visible.
- If the learner revisits a question, accumulate time across visits.
- Prevent double-counting when re-rendering components.
- Store timing in state during the exam and persist to backend periodically (e.g., every 15–30 seconds) to avoid data loss on refresh.

2) Data model (backend)
Create/extend the exam attempt schema to include:
- attemptId, userId, examId, startedAt, submittedAt
- answers: [{ questionId, selectedOptionIds, isCorrect, answeredAt? }]
- timing: {
    perQuestionMs: Record<questionId, number>,
    totalActiveMs: number,         // excludes paused/hidden time
    totalElapsedMs: number         // includes everything from start to submit (optional)
  }

3) Derived metrics (server-side preferred)
Compute and store (or compute on read) the following:
- avgTimePerQuestionMs = totalActiveMs / numberOfQuestionsSeenOrAnswered
  - Define “seen” as a question that was opened at least once; include even if unanswered
- avgTimeCorrectMs and avgTimeIncorrectMs (only among answered questions)
- slowestQuestions: top 5 questionIds by time spent (with ms)
- fastestQuestions: top 5 questionIds by time spent (with ms)
- timeByTopic: aggregate by topic/subtopic (ms and avg seconds)
- pacingScore: compare user avg time/question to exam target pace:
    targetPaceSec = (timeLimitMinutes*60)/numQuestions
    pacingDeltaSec = userAvgSec - targetPaceSec

4) UI: Mock exam review + report card
In the “Report Card” screen for a mock exam attempt, show:
- Total active time (mm:ss)
- Average time per question (seconds, 1 decimal)
- Target pace (seconds per question)
- Pacing delta (faster/slower)
- Average time on correct vs incorrect questions
- A small table/list:
   - “Slowest 5 questions” (question number, topic, time, correct/incorrect)
   - “Fastest 5 questions”
- A “Time by topic” summary (top 5 topics by time)
- In the per-question review screen, display “Time spent on this question” and optionally a pace indicator.

5) Edge cases
- If user submits without opening some questions, exclude unopened questions from avgTimePerQuestionMs OR clearly label metric as “per seen question”. Implement as “per seen question” and show denominator on the report card (e.g., “Avg time per seen question (72/100 seen)”).
- If totalActiveMs is 0 (very fast submit), avoid divide-by-zero; display “N/A”.
- Handle refreshed sessions: attempt resumes should restore perQuestionMs and current question timing state.

6) Testing
Add unit tests for:
- time accumulation across revisits
- pause/resume behavior on visibility change
- correct calculation of avgTimePerQuestionMs, topic aggregates, slowest/fastest
Add an integration test or simple dev harness that simulates navigation events and verifies stored timing.

7) Output
Provide:
- Updated TypeScript types
- Backend migration (if using DB) and updated API routes
- Front-end hooks/components (e.g., useQuestionTimer hook)
- Updated report card UI components
- Brief README notes on how timing works and how to extend metrics

Implement now in the existing codebase with minimal disruption and clear comments.
