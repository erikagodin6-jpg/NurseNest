GOAL
Find and fix the exact reason the /en route renders a blank white screen and fails to paint content.

WORKFLOW

PHASE 1 — REPRODUCE THE FAILURE
1. Start the app locally in the Replit environment.
2. Open the /en route specifically.
3. Inspect:
   - browser console
   - terminal/server logs
   - network tab
   - build output
4. Record the FIRST fatal error that occurs. Prioritize the earliest root error, not follow-on errors.

PHASE 2 — CHECK CORE APP BOOT
Verify all of the following:

1. index.html contains the expected mount node:
   <div id="root"></div>

2. main client entry file correctly mounts React:
   - main.tsx, main.jsx, or equivalent
   - ReactDOM.createRoot(document.getElementById("root")!)
   - renders <App />

3. The App component actually returns valid JSX on first load.
4. No top-level provider, router, theme wrapper, auth wrapper, i18n wrapper, or analytics wrapper is throwing before render.

PHASE 3 — FIND THE EXACT CRASH SOURCE
Check the most likely white-screen causes in this order:

A. ROUTER / /en ROUTE FAILURE
- Verify the /en localized route resolves correctly
- Check whether locale parsing, i18n initialization, or route guards are throwing
- Confirm /en does not depend on missing locale data
- Confirm language loader does not return undefined and then get mapped or destructured unsafely

B. TOP-LEVEL NULL / UNDEFINED DATA CRASH
Look for code that does any of these before data exists:
- array.map(...)
- object.property access
- destructuring from undefined
- Object.entries(undefined)
- reading nested translation keys from missing locale objects
- accessing counts or content objects before load

Add safe guards where necessary:
- optional chaining
- null checks
- fallback empty arrays/objects
- fallback loading or error UI

C. BROKEN IMPORT / MODULE FAILURE
Check App.tsx and all direct imports used on the /en page for:
- bad file paths
- circular imports
- server-only modules imported into client code
- components exported as named but imported as default, or vice versa
- deleted components still referenced
- CSS or asset imports that fail bundling

D. ENVIRONMENT VARIABLE MISUSE
In client code, search for any use of:
- process.env
- undefined API keys
- missing VITE_ variables
- code assuming secrets are available in the browser

Replace incorrect frontend env access with:
- import.meta.env.VITE_...

Add safe fallbacks so the homepage does not crash if a nonessential env var is missing.

E. SSR / WINDOW / DOCUMENT MISUSE
Search for browser-only APIs being called at module scope or in shared code:
- window
- document
- localStorage
- sessionStorage
- matchMedia

If found, move them into guarded client-only logic:
- typeof window !== "undefined"
- useEffect where appropriate

F. INVALID LOCALE / META / HEAD MANAGEMENT
Because the broken route is /en, inspect:
- locale wrapper
- hreflang generation
- canonical generation
- route metadata builder
- title/meta management components

Any exception in head/meta generation must fail safely and never block rendering.

PHASE 4 — TEMPORARY ISOLATION TEST
If the crash source is not obvious, isolate it systematically:

1. Render the simplest possible /en page shell.
2. Re-enable top-level providers one at a time:
   - router
   - i18n
   - auth/session provider
   - theme/provider
   - analytics
   - page sections/components
3. Stop when the crash reappears.
4. Fix the exact failing component or provider.

PHASE 5 — HOMEPAGE SECTION AUDIT

- hero component crash
- translated text object mismatch
- stats/count cards expecting missing numbers
- profession cards expecting missing arrays
- FAQ schema or JSON-LD generation throwing
- dynamic menu or mega-menu data mismatches
- image components with broken required props
- content blocks pulling from empty placeholders

If one section is causing the crash, do NOT remove the whole page permanently.
Fix the data contract or add safe fallback rendering.

PHASE 6 — BUILD AND RUNTIME VALIDATION

PHASE 7 — FAIL-SAFE PROTECTION

Do NOT use these to hide the real bug without fixing it.
