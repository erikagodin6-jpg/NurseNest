Audit the entire NurseNest production site and fix localization so that every user-facing piece of content translates correctly, with French implemented first as the highest priority, then Spanish second.

Primary goal
1. French must be fully working across the entire site before moving on to Spanish.
2. After French is complete and verified, apply the same fixes and coverage to Spanish.
3. Do not leave any mixed-language pages, untranslated strings, fallback English fragments, broken layout, or hardcoded text anywhere in the UI.

Scope
Audit and fix all user-facing content across:
- main site
- pricing pages
- tier pages: RPN, RN, NP
- allied health pages
- new grad pages
- lessons
- flashcards
- question banks
- CAT/mock exams
- rationale sections
- study tools
- dashboards
- onboarding
- auth pages
- account/subscription pages
- checkout-related text that is site-controlled
- search
- nav bars
- footer
- hero sections
- CTA buttons
- banners
- popups
- modals
- forms
- validation messages
- empty states
- loading states
- toast messages
- admin-controlled public content
- blog/public SEO pages
- FAQ
- legal/support/help pages
- email/newsletter signup UI
- all public cards, pills, tabs, badges, filters, dropdown labels, breadcrumbs, table headers, pagination, and tooltips

Core requirements
- Find every hardcoded English string in the front end and replace it with proper i18n keys.
- Ensure translations work for text coming from:
  - static UI components
  - CMS/admin-generated content
  - lesson templates
  - exam generators
  - flashcard templates
  - blog templates
  - category metadata
  - SEO metadata where appropriate
- Preserve brand names, exam names, acronyms, medical abbreviations, and proper nouns where they should not be translated.
- Do not translate product names, Stripe-required identifiers, internal route slugs, database enum values, or system constants unless there is a separate localized display label.
- Maintain the NurseNest design, spacing, typography, and visual hierarchy in all languages.
- Ensure French text displays properly with accents and special characters.
- Ensure Spanish text displays properly with accents and punctuation.
- Prevent text overflow, clipped buttons, broken cards, wrapping issues, or layout shift caused by longer translated strings.

Implementation order
Phase 1: French
- Audit every page/component for untranslated or partially translated content.
- Fix the localization architecture first so coverage is scalable.
- Complete French translations site-wide.
- Add missing keys and normalize inconsistent naming.
- Verify language switching works cleanly with no stale cached English content.

Phase 2: Spanish
- After French passes audit, complete Spanish using the same structure.
- Reuse normalized keys from the French audit.
- Ensure Spanish has the same coverage as French.

Technical requirements
- Use one centralized i18n strategy consistently across the app.
- Eliminate duplicate or conflicting translation sources.
- Standardize translation namespaces by feature area.
- Add missing keys for all UI text.
- Replace inline literals in components with translation lookups.
- Support interpolation safely for dynamic values such as counts, names, plan lengths, prices, question totals, and dates.
- Handle pluralization correctly in English, French, and Spanish.
- Handle fallback logic correctly:
  - preferred language -> target locale
  - fallback -> English only when a key is truly missing
- Log all missing translation keys in development and create a report of what was fixed.
- Create a reusable helper for localized labels coming from enums or database-driven content.
- Ensure SSR/CSR hydration does not cause flash of English before translated content loads.
- Persist user language preference correctly.
- Make sure routes, search state, auth state, and dashboard state do not reset incorrectly when language changes.

Admin/content generation requirements
- Audit whether admin-created content supports localization fields properly.
- Where content is generated from templates, ensure template strings are localized.
- If content records only store English source text, create a safe display-layer approach so all site-controlled framing text still localizes.
- Do not corrupt existing production content.
- Do not mass-translate clinical content fields blindly if that risks poor quality or broken formatting. Instead localize the site-controlled UI first and clearly separate content fields that need future professional translation workflows.
- If any lesson, blog, flashcard, or question explanation is already intended to be multilingual, make sure those render correctly per locale and fall back safely.

SEO and metadata
- Ensure French and Spanish pages output correct localized title tags, meta descriptions, og tags where supported, and hreflang implementation.
- Verify canonical strategy is correct and not causing localized pages to collapse into English indexing.
- Make sure language-specific sitemaps or hreflang references are valid if already supported by the project structure.
- Do not break existing indexed URLs.

Quality control
Create and run a localization audit that:
- crawls all major site routes
- identifies hardcoded strings
- identifies missing translation keys
- identifies pages rendering mixed English/French or English/Spanish
- identifies broken interpolation or placeholder leakage like {{count}}
- identifies layout issues caused by translation length
- identifies untranslated alt text, aria labels, form labels, button labels, and headings
- identifies untranslated mobile navigation and responsive states

Then fix the issues and produce:
1. A summary of what was audited
2. A list of files/components fixed
3. A list of missing keys added
4. A list of pages still blocked from full translation, if any
5. A final readiness report for French
6. A final readiness report for Spanish

Guardrails
- Do not remove existing functionality.
- Do not change pricing logic, auth logic, Stripe logic, or database behavior unless required for localization support.
- Do not break route structure.
- Do not invent fake translations; use clean, natural professional French first, then clean, natural professional Spanish.
- Keep nursing/clinical terminology appropriate and professional.
- Preserve all icons, components, branding, and current design system.
- Test both desktop and mobile.

Definition of done
French is considered complete only when:
- no visible hardcoded English remains on major user-facing pages
- language switcher works everywhere
- navigation, pricing, dashboards, tools, lessons, and public content frames display properly in French
- no major overflow or layout issues remain

Only then complete Spanish to the same standard.

Start by performing the audit, then fix French first, then Spanish, then provide a concise implementation report with remaining gaps if any.
