Audit and optimize the MOBILE performance of the public homepage at /en without changing the visual design, routing structure, tier menus, paywalls, or content hierarchy.

Context:
- This is a Vite + React production site.
- Desktop performance is not the issue here.
- The problem is MOBILE Lighthouse/PageSpeed performance on slow 4G, especially LCP.
- Current mobile report shows roughly:
  - Performance: 65
  - FCP: 3.4s
  - LCP: 7.1s
  - TBT: 150ms
  - CLS: 0.001
- Accessibility, Best Practices, and SEO are already relatively strong.
- We need SAFE optimizations only. Do not introduce risky chunking changes that could recreate the previous React/runtime crash.
- Do not break production. Do not alter branding, translations, navigation logic, or pricing/paywall behavior.

Primary goal:
Improve MOBILE LCP and initial render speed on /en.

Do the following in order:

1. Identify the exact LCP element on the /en homepage
- Determine whether the LCP is the hero heading, hero image, hero card, or another above-the-fold element.
- Report the exact element and why it is slow.

2. Optimize above-the-fold mobile rendering
- Ensure the hero section renders as quickly as possible on mobile.
- If there is a hero image, convert it to an optimized modern format if needed and serve an appropriately sized version for mobile.
- Preload the single most important hero asset only if it is truly above the fold.
- Preload only the font(s) used in the hero section if they are custom fonts and above the fold.
- Add font-display: swap where appropriate.
- Do not preload unnecessary assets.

3. Reduce initial JavaScript on mobile
- Audit what is included in the initial homepage bundle.
- Lazy load non-critical homepage sections below the fold.
- Lazy load heavy route-level components that are not needed for the first screen.
- Do not load analytics-heavy or dashboard-related code on the initial homepage render unless essential.
- Do not use aggressive manual chunk splitting that could recreate dependency ordering issues.
- Prefer safe route/component lazy loading with Suspense boundaries.

4. Reduce unused code on the homepage
- Remove or defer libraries not needed for the first paint on /en.
- Audit large icon imports and replace broad imports with narrower imports where possible.
- Check for heavy animation, charts, or interactive modules being pulled into the homepage bundle unnecessarily.

5. Make mobile animations compositor-friendly
- Audit homepage animations and transitions.
- Replace layout-triggering animations with transform/opacity-based animations where possible.
- Reduce or disable non-essential decorative animations on small screens only if needed for performance.
- Preserve the overall NurseNest look and feel.

6. Eliminate image sizing issues
- Fix any images with incorrect aspect ratio warnings.
- Ensure width and height or aspect-ratio are explicitly defined so layout remains stable.
- Preserve visual appearance.

7. Fix console/browser errors seen by Lighthouse
- Check production console output for /en.
- Resolve any browser errors or missing source issues that affect Lighthouse best practices or runtime reliability.
- Confirm the server does not return HTML for missing JS assets.

8. Preserve SEO and UX
- Do not remove meaningful content from the homepage.
- Do not hide important copy just to improve scores.
- Keep headings semantically correct and in proper order.
- Keep all existing SEO-critical content indexable.

9. Accessibility cleanup while touching the page
- Fix any low-contrast text introduced by pastel styling if it fails accessibility.
- Ensure heading levels are sequential.
- Do not redesign the site.

10. Deliverables
After making the changes, provide:
- a concise summary of exactly what was changed
- what the LCP element was
- what was deferred/lazy loaded
- any assets preloaded
- any mobile-only optimizations added
- any risk notes
- before/after Lighthouse results if available
