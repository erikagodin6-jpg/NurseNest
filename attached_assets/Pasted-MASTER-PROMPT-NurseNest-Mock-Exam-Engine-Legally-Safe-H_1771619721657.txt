MASTER PROMPT — NurseNest Mock Exam Engine (Legally Safe + High Fidelity)
Act as a senior psychometric exam designer, nurse educator, and clinical reasoning specialist building a mock examination engine for a professional nursing education platform called NurseNest.
Audience: Nursing learners preparing for high-stakes licensing examinations (practical nurse, registered nurse, advanced practice).
Critical constraints:
• Generate entirely original questions
• Do NOT reproduce, paraphrase, or imitate known exam items
• Do NOT claim equivalence to any real licensing exam
• Do NOT copy proprietary structures or scoring systems
• The goal is to simulate cognitive experience, not replicate protected exams
Design objectives:
Create a mock exam system that mimics the psychological and cognitive experience of licensing exams while remaining fully independent and original.
 
EXAM EXPERIENCE REQUIREMENTS
The mock exams should resemble real licensing experiences in:
• Question difficulty progression
• Clinical reasoning emphasis
• Prioritization logic
• Safety-critical decision making
• Scenario-based structure
• Timing pressure
• Mixed cognitive load
But must remain clearly original.
 
QUESTION DESIGN RULES
Questions must:
• Be clinically realistic and scenario-driven
• Test reasoning, not trivia recall
• Include plausible distractors reflecting common cognitive errors
• Vary difficulty (easy → moderate → difficult → high-judgment)
• Avoid obvious keyword-triggered answers
• Emphasize prioritization, risk recognition, and mechanism logic
Allowed styles:
• Single best answer
• Clinical prioritization
• SATA-style multi-select
• Clinical judgment / unfolding logic
• Mechanism interpretation
• Lab / assessment integration
Do NOT mimic proprietary phrasing patterns.
 
DIFFICULTY ADAPTATION LOGIC (CRITICAL)
The system must:
• Adjust exam emphasis based on user weakness patterns
• Increase question density in poorly performing systems
• Preserve randomness to prevent predictability
• Avoid making adaptation obvious to users
Example logic:
If learner struggles with cardiovascular → subtly increase hemodynamic, perfusion, and pharmacology-cardiac items.
 
TIMING SIMULATION
Simulate realistic exam pressure:
• Configurable exam lengths (short / standard / intensive)
• Time constraints creating moderate cognitive stress
• Do NOT claim official timing equivalence
 
SCORING & PERFORMANCE MODEL
Instead of claiming “same passing criteria,” use:
• Competency-style performance bands
• Safety & reasoning thresholds
• Domain strength indicators
• Weakness mapping
Avoid claims implying regulatory equivalence.
Safe phrasing example:
“Performance reflects NurseNest competency benchmarks designed to model clinical reasoning demands.”
 
RATIONALE & FEEDBACK ENGINE
After exam completion:
Provide deep explanations including:
• Why correct answers are correct
• Mechanistic reasoning
• Why distractors fail
• Cognitive traps involved
• Clinical safety implications
Avoid simplistic rationales.
 
ANXIETY-REDUCTION DESIGN GOALS
Mock exams should psychologically:
• Reduce fear of unpredictability
• Normalize exam-style thinking
• Build confidence through exposure
• Reinforce reasoning frameworks
 
MONETIZATION MODEL
Design exams as standalone purchasable products:
• Allow non-subscribers to purchase mock exams
• Preserve perceived premium value
• Avoid framing as “cheap alternative”
• Position as high-value assessment experience
 
OUTPUT REQUIREMENTS
Provide:
1.	Exam engine logic
2.	Question generation rules
3.	Difficulty adaptation model
4.	Feedback architecture
5.	UX behavior principles
6.	Safely worded scoring framework
Do not reference real exam ownership or claim equivalence.
 
WHY THIS PROMPT WORKS (Critical Insight)
Most people accidentally prompt AI into legal danger territory by saying:
❌ “Make exams identical to NCLEX”
❌ “Use same passing criteria”
This is risky and unnecessary.
What you actually want is:
✔ Cognitive fidelity
✔ Psychological realism
✔ Clinical reasoning similarity
✔ Original IP-safe content
Licensing exams test thinking patterns, not fixed templates.
You can mirror cognition without copying structure.

Prompt: NurseNest Mock Exam Post-Exam Report Engine
Act as a senior psychometrician, nursing educator, and product analytics designer building the post-exam performance report for a professional nursing education platform called NurseNest.
The platform delivers independently developed mock exams that simulate the cognitive and psychological experience of licensing exams while using original content. Do not reference or claim equivalence to any official exam scoring algorithm.
Primary Objective
Design a post-exam report that:
•	Reduces learner anxiety through clear explanations of performance
•	Produces actionable study priorities
•	Identifies patterns of weakness and strength
•	Quantifies readiness using NurseNest competency bands
•	Drives learning plan recommendations and (optionally) product upsells without being salesy
Inputs Available
Assume the system has these data available:
•	Total exam time allowed and time used
•	Per-question timestamps (start time, end time, revisits)
•	Question metadata: system/topic tags, difficulty rating, question type, cognitive skill tested
•	User responses: selected answers, correctness, confidence rating (if collected), changed answers
•	Item statistics: percent correct across users, discrimination proxy, average time per item
•	Exam blueprint weights (topic distribution targets)
•	User historical performance across systems/topics
 
Report Requirements (Must Include These Sections)
1) Exam Summary (High-level)
Include:
•	Exam type (NurseNest mock exam label)
•	Date/time
•	Total questions completed
•	Total time used vs allowed
•	Overall performance band (NurseNest Readiness Level) with brief explanation
•	“Next step recommendation” in one sentence
2) Time & Pacing Analytics (Critical)
Provide:
•	Average time per question
•	Time distribution (fast, typical, slow)
•	Longest time spent on a single question
•	Questions with multiple revisits
•	Pacing consistency score (stable vs erratic pacing)
•	“Time risk flags” such as:
o	too slow overall
o	too fast (possible guessing)
o	time spikes on specific systems/topics
•	Recommend a pacing target and strategy
3) Difficulty Performance Profile
Provide:
•	Performance by difficulty level (easy / medium / hard / high-judgment)
•	“Hardest questions” list and why they were hard (topic, skill, trap type, time)
•	“Easiest questions” list and why they were easy (mastery signal)
•	Identify if the user is missing easy questions (high-risk indicator)
4) Domain & System Breakdown (Core Value)
Provide:
•	Performance by body system (cardio, resp, neuro, renal, etc.)
•	Performance by topic clusters (e.g., perfusion, gas exchange, acid–base)
•	Performance vs blueprint targets (if the exam is weighted)
•	Strengths: top 3 domains with clear evidence (accuracy + time efficiency)
•	Weaknesses: top 3 domains with clear evidence (accuracy + time cost + error patterns)
5) Clinical Reasoning / Cognitive Skill Breakdown
Tag each question with a cognitive skill and report:
•	Prioritization & safety
•	Assessment interpretation
•	Pathophysiology mechanism reasoning
•	Pharmacology reasoning
•	Communication/escalation judgement
•	Delegation/scope reasoning (where relevant)
Then:
•	Show user performance per skill category
•	Identify which skill failures are most dangerous for exam readiness
6) Error Pattern Analysis (Psychometrics-Style)
Generate insights such as:
•	Common trap patterns the user fell for (e.g., treating symptoms not cause, unsafe first action, missing red flags)
•	“Changed answer analysis” (how often changes helped vs hurt)
•	Overthinking marker: high time + wrong
•	Guessing marker: very low time + wrong
•	Misread marker: fast wrong on easy items
•	“Knowledge gap vs reasoning gap” classification for missed questions
7) Readiness Level + Confidence Calibration
Provide:
•	NurseNest Readiness Level (e.g., Not Ready / Developing / Borderline / Likely Ready / Strongly Ready)
•	Confidence calibration if confidence data exists:
o	Overconfident errors vs underconfident correct answers
If confidence not collected, infer cautiously using timing + difficulty signals.
8) Personalized Study Plan (Actionable Next Steps)
Generate a plan with:
•	Top 3 priority systems/topics to focus on
•	Specific learning activities recommended:
o	lessons to review
o	targeted question sets
o	spaced repetition schedule
o	short daily drills
•	Suggested next mock exam timing (not a guarantee; guidance only)
•	“If you have 3 days / 7 days / 14 days” options
9) Progress Tracking & Comparisons (If history exists)
Show:
•	Trend vs previous exams
•	Areas improving
•	Areas regressing
•	Consistency changes (pacing + accuracy)
•	Suggested goal for next attempt
10) Exportable Report Output
The report must be:
•	Viewable in-app
•	Exportable to PDF
•	Easy to print
Include a compact “one page summary” view.
 
UI/UX Requirements

•	Provide “what this means” tooltips for key metrics
•	Present critical insights first, details later

3) Stripe safety patterns (chargeback + double-grant protection)
Non-negotiable Stripe rules
1.	Grant access only via webhook, not client-side success page.
2.	Verify webhook signature.
3.	Make processing idempotent (never grant twice for the same event).
Recommended Stripe flow (Replit-friendly)
•	Use Stripe Checkout for one-time products (Cram Pack, exam packs).
•	Use Stripe Subscriptions for monthly tiers.
•	Webhooks:
o	checkout.session.completed (for one-time purchases)
o	invoice.payment_succeeded (for subscriptions)
o	customer.subscription.updated / deleted
Idempotency pattern
Create a table stripe_events:
•	event_id (unique)
•	processed_at
•	type
Webhook logic:
•	If event_id already exists → return 200 OK (do nothing)
•	Else → process → insert event_id → commit
Entitlement record pattern (single source of truth)
entitlements table:
•	user_id
•	access_type (none | cram_pack | subscriber_rpn | subscriber_rn | subscriber_np)
•	access_expiry (nullable; for cram)
•	subscription_status (active/canceled/past_due)
•	created_at, updated_at
For exam credits:
exam_credits table:
•	user_id
•	product (mini/mock/intensive)
•	credits_remaining
Refund/chargeback handling (must exist)
•	If Stripe signals refund or dispute:
o	revoke cram pack entitlement (or set expiry to now)
o	freeze exam credits added by that purchase
This reduces disputes and abuse.
