URGENT LESSON PAGE CLEANUP + TAXONOMY REPAIR

Problem:
When asked to switch lessons to the published database, the implementation did the wrong thing.

Instead of inserting lesson content into the correct existing lesson systems/categories, it exposed raw database lessons through a new public-facing category/block and left a large number of unfinished, malformed, duplicate, or unwanted lessons visible in the paid tier.

This is not acceptable.

Goal:
Keep the published database as the content source, but restore a clean, curated, production-quality lesson experience where lessons appear only in their intended systems/categories and unfinished/internal/raw content is hidden from users.

NON-NEGOTIABLE RULE:
The database is the source of lesson content.
It is NOT the source of public page structure.
Public structure must be controlled by curated taxonomy, visibility rules, and publish-quality filters.

-------------------------------------
SECTION 1 — REMOVE THE WRONG PUBLIC DISPLAY
-------------------------------------

1. Remove the accidental public-facing database dump block/category such as:
- “Database Lessons”
- raw catch-all lesson lists
- any fallback section that exposes uncurated lesson inventory to users

2. Do not show raw database records directly on the lessons page.

3. Do not create new public lesson systems/categories unless explicitly approved.

-------------------------------------
SECTION 2 — RESTORE CURATED LESSON TAXONOMY
-------------------------------------

4. Rebuild the lesson rendering logic so lessons appear inside the correct intended systems/categories only.

Examples:
- Neurological lessons go in Neurological
- Endocrine lessons go in Endocrine
- Pediatrics lessons go in Pediatrics
- Maternity lessons go in Maternity
- etc.

5. Existing curated systems are the required public structure.
Map database lessons into those systems instead of exposing them as raw inventory.

6. If a lesson cannot be confidently mapped to an approved system, do NOT show it publicly.
Instead send it to an internal review/unassigned state.

-------------------------------------
SECTION 3 — HIDE UNFINISHED / BAD / INTERNAL CONTENT
-------------------------------------

7. Remove from paid-tier public display any lessons that are:
- unfinished
- placeholder
- malformed
- duplicate
- thin/low-quality
- internal draft
- seed/test content
- badly titled
- inconsistent in naming
- missing required lesson fields
- clearly generated but not production-ready

8. Specifically block lessons with problematic naming patterns such as:
- raw internal labels
- malformed capitalization
- duplicate basics/standalone variants unless intentionally approved
- titles with obvious internal suffixes or tier suffixes that should not show publicly
- titles that are too vague, too raw, or not student-facing

9. Create or enforce a proper publish-quality filter such as:
- published = true
- visibility = public
- quality_status = approved
- has required summary/content/meta
- assigned approved system
- assigned correct tier
- not flagged duplicate
- not flagged internal

If these fields do not exist, add a production-safe equivalent.

-------------------------------------
SECTION 4 — CLEAN UP TITLES AND DUPLICATES
-------------------------------------

10. Audit lesson titles across the paid tier and identify:
- duplicates
- near duplicates
- raw/internal titles
- inconsistent suffixes like “Rpn” where they should not display publicly
- inconsistent “Basics” versus full lesson duplication
- repeated concepts across multiple systems unless intentionally cross-listed

11. Normalize public-facing lesson titles so they are clean, student-friendly, and consistent.

Examples of cleanup goals:
- no raw internal suffixes
- no ugly capitalization
- no obvious admin naming
- no repeated lesson with tiny naming differences unless one is intentionally separate

12. Preserve content records, but clean the PUBLIC display layer and merge/hide duplicates where appropriate.

-------------------------------------
SECTION 5 — FIX PAID TIER EXPERIENCE
-------------------------------------

13. Paid users should see a clean premium library, not a content dump.

14. The page must prioritize:
- curated systems
- polished lesson cards
- correct descriptions
- correct counts
- consistent difficulty/time labels
- no internal junk exposed

15. Remove any long list of things that the user does not want visible publicly.

16. Keep the polished top-of-page systems and curated sections as the primary UX.

-------------------------------------
SECTION 6 — CORRECT DATA FLOW
-------------------------------------

17. Refactor the lesson page data flow to follow this order:

A. fetch database lessons
B. validate lesson records
C. filter by publish-quality rules
D. map to approved taxonomy/system
E. sort within each system
F. render only approved public lessons

NOT:
A. fetch database lessons
B. dump directly to screen

18. If the page currently mixes static systems + raw dbLessons + fallback content, refactor so the final rendered output comes from one clean normalized pipeline.

-------------------------------------
SECTION 7 — ADMIN / REVIEW SAFETY
-------------------------------------

19. Create an internal-only state for lessons that are:
- unassigned
- unmapped
- draft
- needs review
- duplicate candidate
- missing metadata

20. These should be visible only in admin/review workflows, never on the public paid lessons page.

21. Add an admin diagnostic summary if needed:
- total lesson records
- public approved lessons
- hidden draft lessons
- duplicate candidates
- unmapped lessons
- lessons by system
- lessons by tier

-------------------------------------
SECTION 8 — PRESERVE GOOD CONTENT
-------------------------------------

22. Do not delete lesson records unless absolutely necessary.
23. Preserve valid lesson content in the database.
24. Focus on:
- correct mapping
- correct visibility
- correct taxonomy
- clean public rendering

-------------------------------------
SECTION 9 — REQUIRED OUTPUT QUALITY
-------------------------------------

25. The public paid lesson library should feel:
- organized
- premium
- intentional
- exam-aligned
- not cluttered
- not auto-generated
- not like an internal database leak

26. Existing finished curated systems should remain the main user experience.

-------------------------------------
SECTION 10 — TESTING
-------------------------------------

27. Verify on the published site, not just preview:
- no accidental “database lessons” dump section
- no raw catch-all category
- unfinished lessons hidden
- duplicate/ugly titles reduced or removed
- lessons appear in proper systems
- paid tier looks polished
- lesson counts still reflect real content accurately
- no broken routing or blank systems

28. Confirm this separately for:
- RPN

- allied if it shares the same lesson architecture
