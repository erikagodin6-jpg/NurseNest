Make the entire website fully translatable. Right now only the hero page is translatable; every page, component, and UI string must use the i18n system.

Scope (must include all):
- All routes/pages: lessons, pre-nursing, anatomy, blog, FAQ, mock exams, dashboard, flashcards, auth pages, pricing/subscribe, settings, checkout, profile, admin (if present), and any other routes.
- All shared components: navbar, footer, cards, modals, forms, buttons, dropdowns, filters, pagination, empty states, error states, toasts, tooltips, loaders, dialogs, and any UI library wrappers.
- All dynamic UI copy: validation messages, alert text, placeholder text, aria-labels, helper text, headings, table column headers, badges, chips, CTAs, and “no results” / “coming soon” text.

Hard requirements:
1) ZERO hard-coded user-facing strings in TS/TSX/JSX/HTML templates (except the brand name “NurseNest”, which must never be translated).
2) Replace all strings with t("...") calls (or the project’s translation function).
3) Create/expand translation resources for ALL supported locales already in the project (do not remove any locales).
4) Ensure translations work on client navigation and refresh. Language selection must persist (existing behavior) and URL-based locale behavior must be preserved if already implemented.
5) Ensure RTL works for Arabic/Farsi/Urdu if those locales exist (keep dir switching working).
6) Ensure per-route SEO/meta (titles/descriptions) are also translatable and use the same i18n resources.
7) Do not change business logic or redesign UI.

Implementation method (must follow):
A) Create an “i18n coverage audit” pass:
   - Search the entire repo for hard-coded strings in TS/TSX/JSX (examples: text nodes, button labels, headings, placeholder=, title=, aria-label=, toast messages, error messages).
   - Also search for UI strings embedded inside arrays/objects/constants (e.g., tabs, nav items, filters, lesson system titles).
   - Produce a checklist of files/components that contain hard-coded strings and then fix them.

B) Define a consistent translation key schema and apply it everywhere:
   - pages.<pageName>.<section>.<label>
   - components.<componentName>.<label>
   - common.<label>
   - errors.<errorName>
   - forms.<formName>.<field>.<label|placeholder|help|validation>
   - seo.<routeKey>.<title|description>

Keys must be stable, readable, and not random hashes.

C) For any dynamic text values (like “Lesson 1 of 10”, “Showing X results”):
   - Use i18n interpolation and pluralization correctly.

D) Make sure translation resources are complete:
   - English must be the source of truth.
   - Other languages must have entries for every key (use existing translations where present; otherwise fill missing keys with best-effort translation or a clearly marked placeholder that falls back to English).

Verification (must do before finishing):
1) Run through each major route in the browser and confirm all UI text changes when language changes.
2) Grep check: confirm no remaining hard-coded strings in UI files (excluding NurseNest).
3) Confirm RTL locales render correctly.
4) Confirm no runtime errors from missing keys.

Deliverables:
- Updated pages/components using t() everywhere
- Updated translation resources covering all keys for all locales
- A short summary listing which files were updated and any remaining intentional non-translated strings (brand name only)
