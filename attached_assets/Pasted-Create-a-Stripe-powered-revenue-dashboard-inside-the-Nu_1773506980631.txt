Create a Stripe-powered revenue dashboard inside the NurseNest admin panel without breaking any existing checkout, webhook, or subscription logic.

Goal:
Build an admin-only billing analytics dashboard that shows subscription and revenue performance for NurseNest tiers (RPN, RN, NP), plus an area to compare total revenue against business/development costs.

Important rules:
- Do not break existing Stripe checkout, pricing, or webhook flows.
- Do not expose Stripe secret keys to the frontend.
- Read Stripe data server-side only.
- Keep the implementation production-safe.
- If any values are unavailable from Stripe directly, calculate them from existing subscription/order data in the database where possible.
- Match the current app style and admin dashboard design language.

--------------------------------------------------
SECTION 1 — ADMIN DASHBOARD PAGE
--------------------------------------------------

Create a new admin page and sidebar/menu item called:

Billing Dashboard

This page should be visible only to admin users.

Place it in the existing admin navigation.

--------------------------------------------------
SECTION 2 — TOP KPI CARDS
--------------------------------------------------

At the top of the Billing Dashboard, add clean summary cards for:

1. Active Subscribers
- total active subscribers
- also break down by tier:
  - RPN
  - RN
  - NP

2. Monthly Recurring Revenue (MRR)
- total estimated recurring monthly revenue
- show a tier breakdown if possible

3. Total Revenue
- total subscription revenue collected
- total one-time revenue if any exists separately

4. Churn
- cancelled subscriptions in the last 30 days
- churn percentage if possible

5. Failed Payments
- count of subscriptions with recent payment failures or past_due status

6. Free vs Paid Users
- total users on free access
- total users on paid plans

--------------------------------------------------
SECTION 8 — CALCULATION RULES
--------------------------------------------------

Use these rules where applicable:

Active Subscribers:
- subscriptions with active status
- optionally include trialing if your system supports it

MRR:
- convert all recurring subscriptions to estimated monthly value
Examples:
  - monthly plan = full monthly amount
  - 3 month plan = amount / 3
  - 6 month plan = amount / 6
  - yearly plan = amount / 12

Churn:
- cancellations in last 30 days / active subscribers at start of period if possible
- if exact denominator is not available, still show cancelled count and note calculation limitations

Revenue:
- use successful payments only
- exclude failed or refunded payments where possible

Failed Payments:
- subscriptions marked past_due, unpaid, or with recent invoice failure
