The platform keeps running out of memory and failing.
This is causing paying users to lose access to features they are actively paying for.
That is unacceptable.

Your mission is to redesign and harden the system so that PAYING USERS ALWAYS HAVE A USABLE EXPERIENCE for the features they are entitled to, even if the system is under memory pressure or one subsystem partially fails.

- unusable premium page

CORE OBJECTIVE
==================================================

Design and implement a MEMORY-RESILIENT, FAILOVER-SAFE, PREMIUM-ACCESS-PROTECTED architecture.

The system must:

==================================================
SECTION 1 — ROOT CAUSE INVESTIGATION
==================================================

Audit the full platform and identify exactly where memory pressure and failures originate.

Investigate all of the following:

A. MEMORY HOTSPOTS
- inspect runtime memory growth
- identify high-memory routes, handlers, modules, and components
- identify large in-memory objects
- identify oversized payloads
- identify full-dataset loading patterns
- identify repeated hydration or serialization
- identify cache growth with no bounds
- identify whether large locale packs, rationale blobs, question datasets, or admin modules are loaded unnecessarily
- identify memory spikes during exam startup, question-bank load, lesson load, and dashboard rendering

B. REQUEST-TIME HEAVY WORK
- inspect whether the app performs expensive work inside live user requests
- detect AI generation, content transforms, PDF/image generation, translation assembly, audit jobs, indexing, or aggregation happening during request handling
- identify synchronous heavy logic that blocks user experience
- identify any work that should be background-only

C. PREMIUM FEATURE DELIVERY PATHS
Audit all paying-user feature paths, especially:
- exams
- practice/question bank
- flashcards
- lessons
- premium dashboards
- subscription-gated study tools

For each one determine:
- what data is loaded initially
- whether too much is loaded at once
- whether full content is loaded when only partial content is needed
- whether failures in one dependency kill the whole route
- whether fallback behavior exists
- whether entitlement logic is fast and reliable

D. FRONTEND BUNDLE / RENDERING PRESSURE
- identify oversized bundles
- identify admin/generator modules leaking into public or premium routes
- identify excessive shared chunks
- identify eager loading of all locales or all tier content
- identify repeated fetch loops
- identify render loops
- identify heavy charts/editors/builders imported into study experiences
- identify hydration bottlenecks and memory-heavy components

E. BACKEND / API / QUERY PRESSURE
- identify endpoints returning too much data
- identify queries selecting too many rows or columns
- identify N+1 patterns
- identify missing indexes
- identify whether full rationale, translation, media, analytics, and question metadata are being bundled together when unnecessary
- identify whether entitlement checks trigger excessive joins or repeated queries
- identify whether session creation is too heavy

F. DEPLOYMENT / STARTUP PRESSURE
- inspect whether app startup loads too much into memory
- inspect whether deploy/start commands execute heavy repair/backfill/migration/content jobs
- inspect whether all services are packed into one runtime that is too large
- identify what should be split or lazy-initialized

==================================================
SECTION 2 — CRITICAL ARCHITECTURAL RULES
==================================================

Implement the following rules across the system:

1. PAYING USERS MUST ALWAYS GET A USABLE MODE
Every premium feature must support:
- full mode
- light mode
- emergency fallback mode where appropriate

2. NO FULL DATASET LOADS ON INITIAL ENTRY
Do not load:
- all exam questions
- all question bank content
- all rationales
- all translations
- all related lessons
- all analytics
- all media
on first route load unless absolutely required

3. ONLY LOAD WHAT THE USER NEEDS RIGHT NOW
Examples:
- exam route loads only session metadata + active question window
- question bank loads paginated results
- lesson page loads summary first, deeper sections lazily if needed
- flashcards load deck metadata first, cards progressively
- rationale loads only when needed, not for every question up front

4. ISOLATE FAILURES
A failure in:
- analytics
- translation fallback
- progress sync
- media
- explanations
- related-content widgets
must NOT break the core paid study feature

5. ENTITLEMENT CHECKS MUST BE FAST AND ROBUST
A valid paying user must not be blocked by:
- brittle client-only checks
- broken role filters
- null state issues
- bad dropdown logic
- expensive or repeated entitlement lookups

6. HEAVY WORK MUST BE REMOVED FROM USER REQUEST PATHS
Move out of live requests:
- AI generation
- PDF generation
- image generation
- batch translation
- audits
- reindexing
- bulk publishing
- sitemap jobs
- repair/backfill work

==================================================
SECTION 3 — MANDATORY FAILOVER DESIGN
==================================================

For every paid feature, implement failover logic.

A. EXAMS
If full exam mode is unhealthy:
- fall back to lightweight session mode
- only load one question or small window at a time
- defer rationale/results-heavy extras
- preserve answer submission and progress
- keep the exam usable

B. QUESTION BANK / PRACTICE
If full dataset or filters are too heavy:
- load paginated questions only
- disable non-essential enrichments temporarily
- keep answering, scoring, and navigation working

C. FLASHCARDS
If full deck load is too heavy:
- load cards progressively
- preserve study function
- defer media or extras

D. LESSONS
If full lesson content is too heavy:
- load core educational content first
- defer related widgets/media/extras
- keep the lesson readable

E. PREMIUM DASHBOARDS
If analytics/progress calculations are heavy:
- show a simplified version
- use cached summaries if needed
- do not block access to study tools

==================================================
SECTION 4 — IMPLEMENTATION REQUIREMENTS
==================================================

Implement the following production-safe fixes:

A. MEMORY PROTECTION
- remove unbounded caches
- set bounds on in-memory collections
- eliminate oversized request payload assembly
- reduce startup memory
- lazy initialize expensive modules
- prevent heavy public route imports
- reduce serialized object size
- avoid loading all locales/content/tier datasets at once

B. PAYLOAD REDUCTION
- narrow API responses
- only select needed fields
- defer rationale, media, analytics, related content, and extra metadata
- paginate aggressively where needed
- move to windowed loading for exams and large banks

C. ROUTE RESILIENCE
- add route-level error boundaries
- add timeout handling
- add retry-safe data fetching
- prevent infinite loaders
- prevent blank page states
- ensure fallback UIs exist for premium routes
- ensure one broken widget cannot collapse the full feature

D. ENTITLEMENT HARDENING
- preserve all plan/tier gating
- RPN cannot access RN/NP-only premium content unless entitled
- RN cannot access NP-only premium content unless entitled
- valid subscribers must never be falsely denied
- navigation must not hide legitimate study areas incorrectly
- access checks should fail closed only for truly unauthorized users, not for broken state

E. DATABASE / API OPTIMIZATION
- add needed indexes
- remove N+1 patterns
- parallelize safe independent lookups
- reduce join weight
- cache hot read paths where safe
- shape lightweight contracts for premium study routes
- avoid full content hydration at session start

F. BACKGROUND JOB SEPARATION
Move all non-user-critical heavy work out of live requests and out of live route rendering.

G. OBSERVABILITY
Add:
- memory diagnostics
- premium-route failure logging
- slow-query logs
- payload size warnings
- startup timing logs
- route degradation logs showing when fallback mode activates
- clear tracing for why a premium route failed or degraded

SECTION 5 — PREMIUM FEATURE GUARANTEE SYSTEM
==================================================

Create a formal reliability layer for paid features.

Implement a "Premium Feature Guarantee" mechanism with these behaviors:

1. HEALTH CHECKS
Before loading a premium feature, assess whether:
- primary endpoint is healthy
- payload size is acceptable
- memory-safe mode is required

2. MODE SWITCHING
Premium features should support:
- FULL mode
- LIGHT mode
- FALLBACK mode

The app should automatically choose the safest viable mode.

3. GRACEFUL DEGRADATION
If full functionality is unavailable, keep the core educational function available:
- answer questions
- progress through exam/practice
- view core study content
- access entitled materials

4. FAILURE ISOLATION
If one subsystem fails, disable only that subsystem, not the entire study experience.

5. CUSTOMER-SAFE MESSAGING
If degradation occurs, show calm, clear messaging such as:
"You're in a simplified study mode while we optimize performance. Your access remains active."

Do not show panic language.
Do not expose technical errors.
Do not trap the user.
