Big red flag in your narration

“adminFetch will re-parse the body from JSON.stringify …”

That is unusual and often wrong. Most fetch wrappers do not “re-parse” JSON bodies; they pass them through. If you pass body: JSON.stringify(...), the wrapper should not stringify again. If it does, you end up with "\"{...}\"".

✅ Safe approach:

If your wrapper expects an object: pass { prompt, ... }

If your wrapper expects a string: pass JSON.stringify(...)

Do one or the other, not both.

If you want a quick sanity test: log the request body server-side once. If it’s a string containing escaped quotes, you’re double-stringifying.

About the “missing anatomy translation keys” detour

If your site already supports fallback-to-English when a key is missing, then this is not urgent. Do it after the monetization pipeline is stable.

If you must patch it now, do it safely:

Safe strategy (lowest risk)

Add the missing keys to all languages but set values to the English string as a fallback.

Later, replace with real translations.

That avoids breaking runtime while not risking a massive structural edit.

Safer than regex: use ts-morph to update the TS object

If your translations live in a big i18n-translations.ts export object, use AST edits so you don’t corrupt syntax.

Here’s a practical approach (outline + working code skeleton) you can hand to your dev AI:

// scripts/add-anatomy-keys.ts
import { Project, SyntaxKind } from "ts-morph";
import path from "path";
import fs from "fs";

const FILE = path.resolve("src/i18n/i18n-translations.ts");
const project = new Project({ tsConfigFilePath: path.resolve("tsconfig.json") });
const source = project.addSourceFileAtPath(FILE);

// 1) Locate the exported translations object (adjust name as needed)
const varDecl = source.getVariableDeclarationOrThrow("translations");
const obj = varDecl.getInitializerIfKindOrThrow(SyntaxKind.ObjectLiteralExpression);

// 2) Define the anatomy keys you want everywhere (English fallback values)
const anatomyPatch = {
  anatomy: {
    hero: { title: "Anatomy", subtitle: "..." },
    examContext: { title: "..." },
    faq: { q1: "...", a1: "..." },
    conversion: { title: "..." },
    links: { nclex: "..." },
  }
};

// 3) Helper to deep-merge into each language object
function upsertObjectLiteral(targetObj: any, patch: any) {
  for (const [k, v] of Object.entries(patch)) {
    const existing = targetObj.getProperty(k);
    if (!existing) {
      // add property
      targetObj.addPropertyAssignment({
        name: k,
        initializer: JSON.stringify(v, null, 2),
      });
      continue;
    }
    // if both are objects, recurse; otherwise overwrite
    const init = (existing as any).getInitializer?.();
    if (init && init.getKind() === SyntaxKind.ObjectLiteralExpression && v && typeof v === "object" && !Array.isArray(v)) {
      upsertObjectLiteral(init, v);
    } else {
      (existing as any).setInitializer(JSON.stringify(v, null, 2));
    }
  }
}

// 4) Iterate languages: translations.en, translations.fr, etc.
const langProps = obj.getProperties();
for (const prop of langProps) {
  const langInit = (prop as any).getInitializer?.();
  if (!langInit || langInit.getKind() !== SyntaxKind.ObjectLiteralExpression) continue;
  upsertObjectLiteral(langInit, anatomyPatch);
}

// 5) Save
source.saveSync();
console.log("Patched anatomy keys into all languages with English fallback.");

This avoids “find line numbers and insert text” risks.
