Option B: Turn it into a strict guard
export function premiumRouteGuard(requiredTier?: "rpn"|"rn"|"np") {
  return (req, res, next) => {
    const tier = req.user?.tier ?? "free";
    if (tier === "admin") return next();

    if (!requiredTier) {
      // if you truly have non-tiered premium routes
      if (tier === "free") return res.status(403).json({ error: "Premium only" });
      return next();
    }

    if (tier !== requiredTier) {
      return res.status(403).json({ error: "Tier mismatch", requiredTier, tier });
    }

    next();
  };
}
Then call it with the required tier per route. No more “any paid tier”.
3) Apply exact-tier checks to routes AND to the database query
Even if routes are gated, enforce tier in queries so mistakes can’t leak data.
Example:
// GET /api/qbank/:tier
app.get(
  "/api/qbank/:tier",
  requireAuth,
  (req, res, next) => requireExactTier(req.params.tier as any)(req, res, next),
  async (req, res) => {
    const tier = req.params.tier; // "rpn" | "rn" | "np"

    const questions = await db.questions.findMany({
      where: { tier },        // hard filter by tier
      take: 50,
      orderBy: { id: "desc" },
    });

    res.json({ questions });
  }
);
Do not allow non-admin users to override tier via query/body/header.
4) The bigger issue: if qbank content is embedded in the JS bundle, your paywall is fundamentally bypassable
If your questions are bundled into the client (e.g., imported JSON, arrays, static files under public/, or compiled into assets/*.js), then:
Any logged-in (or even logged-out) user can open DevTools and extract them.
Tier enforcement won’t matter because the content is already shipped.
You need to move premium question data out of the bundle.
Non-negotiable for a real paywall:
Premium content must live server-side (DB or protected object storage)
and be delivered only via authenticated, tier-checked endpoints.
Quick way to confirm:
Open DevTools → Network → filter js → open the biggest bundle → search for a known question stem.
If it’s there, it’s already leaked.

Logging unusual access patterns
6) Fix extractUserTier / resolveTierFromRequest risk
If tier can be resolved from “various auth sources,” that’s often where AI “helpfully” broke things.
Rule:
Tier should come from the server’s user record, not from the request payload, not from headers, not from localStorage.
So: resolveTierFromRequest should only help identify the user (session/JWT), then tier comes from DB.
