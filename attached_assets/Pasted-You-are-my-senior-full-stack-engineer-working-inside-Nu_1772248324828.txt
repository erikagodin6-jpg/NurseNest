PART 2 — DATABASE DESIGN (DO THIS PROPERLY)
====================================================

Implement a normalized translation table:

content_translations
- id (pk)
- content_type (lesson|module|question|flashcard|case|glossary|seo_page|faq|ui_string)
- content_id (string/uuid/int)
- language_code (one of the 15)
- field_name (title|content|meta_title|meta_description|faq_json|stem|options_json|rationale|etc.)
- translated_text (LONGTEXT)
- translation_status (auto|human_reviewed)
- source_hash (hash of the English field at time of translation)
- updated_at

Rules:
- English content remains stored in the primary content tables as the source of truth.
- For any non-English language, translations live in content_translations.
- Never overwrite translation_status=human_reviewed.
- If English changes, source_hash changes => mark translation as stale (needs review).

PART 3 — API: LANGUAGE-AWARE CONTENT FETCHING
====================================================

Every content GET must accept language:
- via URL prefix /{lang}/...
- or Accept-Language header
- or user profile preference

Implement server middleware:
- parse lang from URL prefix
- validate against supported language list
- attach req.lang

Update lesson endpoints:
GET /api/lessons/:slug
Return:
- base English fields
- + translated overrides for req.lang (if exists)
- + translation_status flags (missing/stale)

Same for modules/menu endpoints:
GET /api/modules
GET /api/lesson-categories
Return translated titles/descriptions when available.

Implement a helper:
getTranslatedField(content_type, content_id, field_name, lang):
- if lang == "en" return base
- else return translation if exists
- else fallback to base English

IMPORTANT:
Make sure the FRONTEND uses ONLY these language-aware endpoints, not raw DB fields.
