Investigate and permanently fix the production crash that occurs when a user clicks into an exam.

CURRENT SYMPTOM
When a user clicks an exam, the page loads an error boundary screen that says:
“Something went wrong. An unexpected error occurred. Please try reloading the page.”

This is a runtime failure on the exam route and must be fixed at the root cause, not hidden.

GOAL
1. Find the exact source of the crash
2. Fix it permanently
3. Add defensive protections so one bad exam/question cannot crash the full page
4. Ensure published/live exams always load reliably for customers
5. Add monitoring/logging so this never silently happens again

REQUIRED DEBUG WORK
1. Trace the exact route involved when clicking an exam
   - identify the route component
   - identify the loader/query/API call used
   - identify all child components rendered after navigation

2. Reproduce the failure locally and in production mode
   - click into multiple exams
   - identify whether failure happens for all exams or only some
   - capture the exact stack trace and failing component
   - inspect browser console, network responses, and server logs

3. Audit these likely failure points:
   - exam ID / slug / route param mismatch
   - null or undefined exam object
   - questions array missing or empty
   - malformed question records
   - unsupported question type
   - missing rationale/options/stem/content fields
   - language/i18n field mismatch
   - exam access/subscription guard returning invalid state
   - API response shape drift between backend and frontend
   - lazy import / chunk loading issue
   - recent refactor breaking prop names or expected types

4. Add temporary console and server logs during debugging for:
   - clicked exam ID
   - fetched exam payload
   - question count
   - first question shape
   - user role / entitlement result
   - current locale / language
   - exact failing component name
   Remove noisy logs after fix, but keep structured error logging.

MANDATORY FIXES
A. Route-level safety
- Make the exam page robust against missing data
- Never render child exam components until exam data is validated
- Show a proper loading state while fetching
- Show a user-friendly recoverable state if exam data is unavailable
- Do not throw uncaught runtime errors

B. Data validation
- Add a strict normalization/validation layer before rendering exam data
- Validate:
  - exam id
  - title
  - status
  - questions array
  - question type
  - options
  - correct answer payload
  - rationale fields
  - language fields
- If invalid, log the exact invalid record and skip or gracefully fail that record instead of crashing the page

C. Question renderer hardening
- Wrap each question renderer with guards
- Unsupported or malformed questions must render a fallback card:
  “This question is temporarily unavailable.”
- One bad question must never crash the whole exam

D. API contract alignment
- Ensure backend exam payload exactly matches frontend types/interfaces
- Update TS types/zod schemas/interfaces to match real production data
- Remove any stale assumptions from old exam generator formats

E. Error observability
- Add a centralized error logger for exam load/render failures
- Capture:
  - route
  - exam id
  - user id if available
  - locale
  - failing question id if applicable
  - stack trace
- Surface this in admin diagnostics if possible

F. User protection
- If exam fetch fails, show:
  - “We couldn’t load this exam right now.”
  - retry button
  - return to exams button
- Do not show a blank white page or generic crash screen for recoverable errors

G. Regression protection
- Add tests for:
  - valid exam loads
  - missing exam object
  - empty questions array
  - malformed question record
  - unsupported question type
  - translation field missing
  - access guard edge cases
- Add one integration test covering click from exam list into live exam page

SPECIFIC THINGS TO CHECK FIRST
1. Any recent changes to:
   - exam route component
   - live exam player
   - CAT exam logic
   - question renderer
   - multilingual content handling
   - role/subscription gating
   - API response serializers
   - suspense/lazy imports

2. Any records in the database where:
   - exam is published but has zero valid questions
   - question JSON is malformed
   - question type enum does not match frontend switch cases
   - translated content exists in one field shape but frontend expects another

3. Confirm whether the crash happens in:
   - nursing exams
   - allied health exams
   - CAT exams
   - free exams
   - paid exams
   - only certain locales/languages

IMPLEMENTATION STANDARD
- Fix root cause cleanly
- Do not use try/catch to merely suppress errors without handling them properly
- Do not remove features to make the crash disappear
- Do not hardcode around one exam only
- Make the solution production-safe and scalable

DELIVERABLES
1. Root cause summary
2. Exact files changed
3. Exact fix implemented
4. Any bad database records found and corrected
5. Tests added
6. Confirmation that exam click-through works again across all exam types
