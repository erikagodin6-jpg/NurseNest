Urgent bug fix: the exam engine is still failing with the message “Unable to start exam. Please try again in a moment.”

I need you to diagnose and fix the actual root cause, not just hide the error.

Problem
- When a user tries to start an exam/test bank, the app throws:
  “Unable to start exam. Please try again in a moment.”
- This needs a real end-to-end fix.

Your task
1. Trace the full exam start flow from click to rendered exam session.
2. Identify exactly where it is failing:
   - frontend click handler
   - route/navigation
   - API request
   - auth/session check
   - tier access/paywall check
   - exam generation logic
   - DB read/write
   - exam session creation
   - question fetch
   - malformed payload
   - null/undefined data
   - environment variable/config issue
3. Fix the failure at the source.
4. Add robust logging so this cannot fail silently again.
5. Preserve all existing business rules, including access control and paywalls.

What to inspect
- The Start Exam / Begin Exam / Launch Test Bank button handler
- Any exam initialization function or hook
- API endpoints that create or fetch exam sessions
- Server logs and browser console errors
- DB queries for:
  - exam metadata
  - question pools
  - user access / subscription / entitlement
  - session creation records
- Validation logic for required fields
- Any recent changes to navigation/header/layout that might have broken route state or params
- Any recent changes to lesson/test bank mapping
- Any slug/id mismatch
- Any exam type or tier param mismatch
- Any unpublished or empty test bank content being passed into the exam engine

Likely failure points to check carefully
- Missing or invalid exam ID / bank ID / slug
- Start button pointing to wrong route
- API returning 400/401/403/404/500
- User entitlement check falsely blocking access
- Empty question set causing exam creation to fail
- Server expecting fields the frontend is not sending
- Frontend expecting fields the API is not returning
- Session row failing to insert
- Race condition or async state issue
- Error boundary swallowing the real exception and replacing it with the generic message
- Broken environment/config after recent edits

Required fix behavior
- If the exam has valid content and the user has access, the exam must start successfully.
- If content is missing, show a specific admin-debuggable reason in logs.
- If access is denied, show the correct access/paywall message, not the generic startup failure.
- If a test bank is empty or mislinked, identify and repair the mapping instead of letting start fail.

Logging requirements
Add meaningful logs for:
- clicked start exam
- exam/tier/slug/id received
- user/auth/subscription state
- API request payload
- API response status
- question count found
- exam session creation success/failure
- exact thrown error and stack trace

Do not just log “failed.”
Log the real cause.

UI requirements
- Keep the current exam page design unless a tiny change is needed for functionality.
- Do not redesign the page.
- Do not remove paywalls or entitlement logic.
- Do not hardcode fake success states.
- Do not bypass errors by skipping validation.

Error handling requirements
Replace the single generic failure with more precise handling:
- no questions found
- unauthorized
- subscription required
- invalid exam mapping
- server error
But only after the root issue is fixed.

QA checklist
- Start exam works for valid test banks
- Exam sessions are created consistently
- Questions load properly
- Subscription-protected exams still respect access control
- Empty/misconfigured banks are identified
- No generic startup failure remains without detailed logs
- Works across RPN, RN, NP, allied health, and any tier using the same exam engine

Acceptance criteria
- Clicking Start Exam successfully launches the exam when valid
- The root cause of the current failure is fixed
- The generic “Unable to start exam” error no longer appears for valid exams
- Logs clearly show what failed if there is ever a real issue again
