1. SAVE ANSWERS INCREMENTALLY DURING THE EXAM

Implement or refactor so that each answer is persisted during the exam, not only at the end.

Create or fix endpoint:

POST /api/exams/attempts/:attemptId/answer

Request body:
{
  "questionId": "string",
  "selectedOption": "string",
  "timeSpentSeconds": 12,
  "sequence": 5,
  "clientTimestamp": "ISO_STRING"
}

Backend requirements:
- validate attemptId exists
- validate question belongs to this exam/attempt
- upsert answer by unique key (attemptId + questionId)
- store only minimal answer data
- do not store full question objects in answer rows
- do not recompute the entire exam on every answer save
- return success quickly

Answer-saving must be idempotent:
- repeat save for same question updates existing row safely
- no duplicates
- no attempt corruption

==================================================
2. FINAL SUBMIT MUST BE LIGHTWEIGHT
==================================================

Create or refactor endpoint:

POST /api/exams/attempts/:attemptId/complete

Request body:
{
  "finalQuestionCount": number,
  "clientTimestamp": "ISO_STRING"
}

This endpoint must NOT receive:
- full question set
- rationales
- answer history blobs
- exam metadata duplicates
- large arrays of unnecessary frontend state

This endpoint should only:
- verify attempt exists
- verify attempt is not already finalized
- mark attempt as “completion_requested”
- enqueue or trigger lightweight finalization
- return immediately

Target response time:
- under 2 seconds

Return shape:
{
  "ok": true,
  "attemptId": "...",
  "status": "processing"
}

==================================================
3. MOVE SCORING / FINALIZATION OUT OF THE REQUEST PATH
==================================================

Scoring and finalization must not block the submit request.

Implement async-safe finalization logic:
- load persisted answers from DB
- compute CAT result server-side
- compute score / pass estimate / exam summary
- save only final summary fields
- mark attempt as completed
- record completion timestamp

If background jobs already exist, use them.
If not, implement a safe deferred service layer or queue-like internal worker pattern.

Do not leave the user hanging on one giant synchronous request.

4. ADD RESULT POLLING / RECOVERY
==================================================

GET /api/exams/attempts/:attemptId/result

- in_progress
- processing
- completed
- failed

If completed, return summary/result payload.
If processing, return lightweight status.
If failed, return retryable status without losing data.

Frontend must poll this endpoint after completion request until result is ready.
