You are working on the live NurseNest production system.

The immediate incident response and stabilization work is already underway. Now implement a PERMANENT RECURRENCE PREVENTION SYSTEM so this class of outage does not happen again.

This is a production hardening and operational safety initiative.

The recent incident included:
- repeated memory exhaustion
- emergency mode loops
- heavy routes destabilizing the app
- exam_open_flow failures
- invalid content reaching production
- alert storms and misclassified alerts
- deploy freeze activation

Your objective is to build safeguards so future regressions are caught early, contained automatically, and prevented from reaching users.

==================================================
SECTION 1 — PRE-DEPLOY SAFETY GATES
==================================================

Build strict pre-deploy checks so risky code cannot be promoted safely.

Implement:
- memory regression checks on critical routes
- payload size checks for exam and content endpoints
- smoke tests for exam_open_flow
- content validation checks
- alert template classification tests
- emergency mode logic tests
- fallback rendering tests
- route timeout tests

A deployment must be blocked automatically if:
- memory exceeds safe thresholds in test runs
- exam_open_flow fails
- invalid content is marked publishable
- alert classification is incorrect
- critical fallback paths are broken

Add a clear pre-deploy report showing pass/fail reasons.

==================================================
SECTION 2 — RELEASE GUARDRAILS
==================================================

Implement safer release mechanics.

Add:
- staged rollout / canary release support if available
- deploy freeze rules triggered by active incidents
- post-deploy smoke tests on critical user flows
- automatic rollback or rollback recommendation if key flows fail
- deployment annotations tied to incident timelines

Critical flows to verify after deployment:
- home page
- login/signup
- dashboard
- exam listing
- exam open flow
- question answering
- payment flow
- admin publishing flow

==================================================
SECTION 3 — RESOURCE BUDGETS
==================================================

Define and enforce production resource budgets.

Set explicit budgets for:
- max payload size per endpoint
- max memory growth per request
- max route latency
- max concurrent heavy jobs
- max exam question batch size
- max cached object count / size
- max synthetic monitor frequency on fragile routes

If any budget is exceeded:
- warn in staging
- block deploy if critical
- log exact offender and reason

==================================================
SECTION 4 — ARCHITECTURE ENFORCEMENT
==================================================

Prevent architectural drift.

Implement guardrails so heavy workloads cannot silently move back into the main web process.

Specifically:
- define which jobs are allowed in the web process
- define which jobs must run in workers/queues
- add clear boundaries in code/config
- fail fast if disallowed heavy jobs are invoked in web runtime
- document approved execution paths for AI generation, bulk exam creation, rendering, scheduled audits, and content repair

==================================================
SECTION 5 — CONTENT QUALITY GATES
==================================================

Make invalid content structurally impossible to publish.

Implement:
- draft-only until validation passes
- required schema checks
- minimum valid item counts
- tier/language consistency checks
- publishing transaction that fails atomically if validation fails
- nightly or scheduled integrity audit with automatic quarantine
- admin alerts only for actionable content failures, deduplicated properly

Ensure invalid content cannot appear in public listings even temporarily.

==================================================
SECTION 6 — RESILIENCE TESTING
==================================================

Create deliberate failure drills and resilience checks.

Test and verify system behavior when:
- memory is high
- exam service is slow
- content item is invalid
- DB query is slow or partially fails
- background worker is unavailable
- alert service fails
- external billing or API call times out

Expected behavior:
- graceful fallback
- no blank pages
- no endless retry loops
- no alert storms
- no dead-end for users

==================================================
SECTION 7 — ALERT HYGIENE
==================================================

Implement lasting alert discipline.

Add:
- deduplication windows
- per-alert-type cooldowns
- incident grouping
- severity thresholds
- suppression of noisy low-value repeats
- correct template mapping tests
- operator-friendly summaries instead of floods

Every alert must answer:
- what failed
- how severe it is
- whether users are affected
- what component is affected
- whether this is new or part of an existing incident

==================================================
SECTION 8 — OBSERVABILITY DASHBOARD
==================================================

Create an operational view that makes regressions obvious.

Track and surface:
- RSS / heap trends
- error rate by route
- exam_open_flow success rate
- latency percentiles on critical routes
- payload sizes
- heavy job queue depth
- invalid content count
- alert count by type
- active incidents
- emergency mode activations
- deploy-to-incident correlation

Make it easy to answer:
- what changed
- what route regressed
- what content is broken
- whether the system is recovering

==================================================
SECTION 9 — RUNBOOKS AND AUTO-CONTAINMENT
==================================================

Create runbooks and automatic protective actions.

When memory is critical:
- shed heavy load
- throttle or block high-cost routes
- preserve core user flows

- notify operators once per incident window

- quarantine automatically
